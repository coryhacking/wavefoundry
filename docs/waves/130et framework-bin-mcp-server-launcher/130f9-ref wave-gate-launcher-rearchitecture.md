# Rearchitect `wave-gate` to Match the Standard Bin-Launcher Pattern

Change ID: `130f9-ref wave-gate-launcher-rearchitecture`
Change Status: `implemented`
Owner: Engineering
Status: implemented
Last verified: 2026-05-31
Wave: 130et framework-bin-mcp-server-launcher

## Rationale

`.wavefoundry/bin/wave-gate` is currently a ~95-line standalone Python script that lives entirely in the bin directory. Every other bin launcher (`docs-lint`, `docs-gardener`, `wave-dashboard`, `update-indexes`, `setup-wavefoundry`, `upgrade-wavefoundry`, and the just-landed `mcp-server` from `130eu`) is a thin shell launcher that resolves repo root and venv Python and then `exec`s into a framework script under `.wavefoundry/framework/scripts/`. This pattern makes the launchers:

- Generated reproducibly from `render_bin_launchers()` (no drift; new target repos always get the canonical content)
- Venv-aware via the shared `_venv_block` shell snippet
- Testable as both the launcher (shell) and the implementation (Python script) independently
- Distributable through `build_pack.py` since the implementation lives in the packaged framework dir

`wave-gate` is the outlier. Its logic lives in the bin directory, which means:

- `render_bin_launchers()` cannot generate it without inlining ~95 lines of Python source (and the existing code doesn't even attempt to)
- It is not venv-aware (uses `#!/usr/bin/env python3`, missing the framework venv shell block)
- Its implementation is not visible to framework scripts that might want to call the same logic in-process
- It is invisible to the framework graph index (which only walks `.wavefoundry/framework/scripts/`)

This change rearchitects `wave-gate` to follow the standard pattern: extract the Python logic into `.wavefoundry/framework/scripts/wave_gate.py` and make `.wavefoundry/bin/wave-gate` a thin shell launcher generated by `render_bin_launchers()`.

Source: operator request 2026-05-31 ("Address the wave-gate issue with rearchitecture it so it follows the same pattern as other ones").

## Requirements

1. A new framework script `.wavefoundry/framework/scripts/wave_gate.py` must contain the gate management logic currently in `.wavefoundry/bin/wave-gate`. The CLI surface (subcommands `open`, `close`, `status`; gate names `seed_edit_allowed`, `framework_edit_allowed`; exit codes; stdout/stderr messages) must be byte-identical to the existing launcher's behavior. Operators and the framework MCP `wave_gate_*` tools must see no behavioral change.
2. The new script must resolve the repo root from its own `__file__` location (so it works even when invoked from a non-repo CWD), and must accept an optional `--root <path>` argument that overrides the default. This matches the canonical pattern other framework scripts follow (e.g. `setup_index.py`).
3. `render_bin_launchers()` in `render_platform_surfaces.py` must write `.wavefoundry/bin/wave-gate` as a thin shell launcher (same shape as `mcp-server`, `docs-lint`, etc.) that resolves the venv Python and `exec`s into `wave_gate.py`. The launcher must be executable.
4. Existing tests that reference `.wavefoundry/bin/wave-gate` behavior (none observed in the framework test suite at the time of this change; if any exist they must continue to pass). New tests must cover the launcher generation and the new Python script's CLI surface.
5. The old `.wavefoundry/bin/wave-gate` standalone Python script in this repo will be overwritten by the next `wave_sync_surfaces` invocation with the new shell launcher. This is the desired behavior — the canonical content moves into the framework. Document this so operators are not surprised.
6. The framework graph index must pick up the new `wave_gate.py` symbol on the next rebuild (no special handling required — it lives in the indexed framework scripts directory).

## Scope

**Problem statement:** `wave-gate` is the only bin launcher whose implementation lives in the bin directory rather than in `.wavefoundry/framework/scripts/`. This means it can't be generated by `render_bin_launchers()`, isn't venv-aware, and is invisible to the framework graph index.

**In scope:**

- `.wavefoundry/framework/scripts/wave_gate.py` — NEW file containing the gate management logic (open, close, status subcommands). Resolves repo root from `__file__` with an optional `--root` override.
- `.wavefoundry/framework/scripts/render_platform_surfaces.py` — add `wave_gate_src` template and `write_text(bin_dir / "wave-gate", ...)` call in `render_bin_launchers()` immediately after `mcp_server_src`.
- `.wavefoundry/framework/scripts/tests/test_wave_gate.py` — NEW test file covering the Python CLI: open/close/status behavior, invalid-gate handling, JSON read/write, already-open and already-closed cases. Mirrors the shape of other CLI test files.
- `.wavefoundry/framework/scripts/tests/test_render_platform_surfaces.py` — extend `test_creates_bin_launchers` and `test_bin_launchers_are_executable` to cover the new shell launcher.

**Out of scope:**

- Changing the `.wavefoundry/guard-overrides.json` schema. The new script reads/writes the same JSON format the standalone launcher does.
- Changing the gate name vocabulary. `seed_edit_allowed` and `framework_edit_allowed` remain the only valid gates at this layer; the MCP `wave_gate_*` tools handle `design_system_edit_allowed` separately.
- Refactoring the bin launcher templates into a data-driven loop. Three small repetitions (`mcp_server_src` + `wave_gate_src` added this wave) is still under the threshold where a loop would pay back.
- Touching the MCP `wave_gate_open` / `wave_gate_close` / `wave_gate_status` tools. They already live in `server_impl.py` and operate on the same JSON file — no behavioral change there.

## Decision Log

| Date | Decision | Reason | Alternatives |
|---|---|---|---|
| 2026-05-31 | Move logic to `wave_gate.py` rather than inline into `render_bin_launchers()` | Inlining ~95 lines of Python as a Python string literal would be hostile to readers and lose IDE/linter support; keeping it as a separate file matches every other framework script | Inline as a string in `render_bin_launchers()` (rejected — terrible ergonomics, lossy on linting and indexing) |
| 2026-05-31 | Resolve repo root from `__file__` with an optional `--root` override | Matches the canonical pattern (`setup_index.py`, `setup_wavefoundry.py`); standalone-invocation still works without args; `--root` makes the script testable from anywhere | Require `--root` always (rejected — adds friction for standalone use); compute from CWD (rejected — fragile if invoked from outside the repo) |
| 2026-05-31 | Keep gate name vocabulary unchanged | The CLI surface contract is the headline; widening or changing the vocabulary is a separate concern with its own review | Add `design_system_edit_allowed` here (rejected — MCP layer already handles it; CLI surface is out of scope for vocabulary changes) |
| 2026-05-31 | Land in the same wave as `130eu` (mcp-server) | Both are bin-launcher provisioning fixes from the same operator report; landing together keeps the context hot and the test surface coherent | File as a separate wave (rejected — overhead for a single related change) |

## Risks

| Risk | Mitigation |
|---|---|
| The existing standalone `wave-gate` may have a behavioral nuance the test suite doesn't capture | AC-7 mandates manual smoke verification; byte-equivalent open/close/status round-trip on a temp dir before close |
| Operators who shell-aliased the standalone script may see breakage on next sync | The new launcher accepts the same arguments; the alias continues to work if it points at `.wavefoundry/bin/wave-gate` (which is the same path) |
| The new `wave_gate.py` could conflict with existing framework imports if a module of that name already exists | Verified via `ls .wavefoundry/framework/scripts/` before naming; no conflict |
