## Rationale

Field feedback (Solaris, framework 1.10.1+p9pc) reported reviewer/builder subagents making **zero** MCP `code_*` calls across a 4-lane review fan-out (~145 grep/sed calls, ~115 Reads, 0 MCP). Verification against this repo (2026-07-03) found the load-bearing defect is mechanical, not wording:

1. **Every rendered thin pointer forbids the MCP tools.** All five committed `.claude/agents/` wrappers (guru + four factor reviewers) set `tools: Read, Grep, Glob, Bash`. An explicit allowlist is not additive — a spawned guru or factor subagent **cannot call** `code_ask`/`code_references`/any `mcp__wavefoundry__*` tool. The guru wrapper is self-contradictory: its own body (lines 106-108) instructs it to call `code_ask`, `code_search`, `code_read`, etc., while its frontmatter denies them. Edit points: the hardcoded guru template constant at `render_agent_surfaces.py:94` and the factor-wrapper template in seed `050-agent-entry-surface-bootstrap.prompt.md:211`.
2. **The framework documents a premise these wrappers break.** Seeds `180-implement-feature.prompt.md:133`, `100-project-prompt-surface-bootstrap.prompt.md:81`, and `050-agent-entry-surface-bootstrap.prompt.md:161,546` (and the rendered AGENTS.md auto-Guru section) state "Task-spawned subagents inherit the parent session's MCP tools by default." That is true only for agents *without* an explicit `tools:` allowlist — the framework's own rendered role wrappers all set one. The delegation guidance built on this premise silently fails for exactly the role-typed agents it recommends routing work through.

Downstream repos cannot durably self-fix: the wrappers are regenerated by `render_agent_surfaces.py` on upgrade (seed-050: "Do not hand-edit generated marker regions... change `render_agent_surfaces.py` instead"). This is the prerequisite for the companion wording change (`1p9ql`) — guidance telling agents to use tools they cannot load would reproduce the guru contradiction at scale.

## Requirements

1. **Guru wrapper allowlist.** The `CLAUDE_GURU_AGENT` template (`render_agent_surfaces.py:94`) grants the read-only Wavefoundry MCP retrieval tools alongside `Read, Grep, Glob, Bash`: the `code_*` navigation/query tools (`code_ask`, `code_search`, `code_keyword`, `code_read`, `code_outline`, `code_definition`, `code_references`, `code_callhierarchy`, `code_dependencies`, `code_impact`, `code_list_files`, `code_constants`, `code_pattern`, `code_callgraph`, `code_graph_path`, `code_graph_community`) plus `docs_search` and `seed_get`. No write-capable or lifecycle-mutating MCP tools (`wave_*` mutators) are added — the wrappers stay read-only. Exact frontmatter syntax (explicit enumeration vs a host-supported wildcard) is confirmed against current Claude Code subagent semantics at implementation; explicit enumeration is the safe default.
2. **Factor wrapper allowlist.** The factor-review wrapper template in seed-050 (line 211 region) gains the same read-only retrieval set, with the same no-mutator constraint.
3. **Premise corrections.** Every "subagents inherit the parent's MCP tools by default" statement (seeds 180:133, 100:81, 050:161,546; any other instance found by a tree-wide grep) is corrected to state the real rule: inheritance applies only to agents without an explicit `tools:` allowlist; agents with an allowlist must name the MCP tools in it, and the framework's rendered wrappers do. Wording stays self-contained (no internal change-ID refs in seeds).
4. **Self-hosted surfaces re-rendered.** After the script/seed edits, `wf render-surfaces` (or the equivalent) regenerates this repo's `.claude/agents/*.md`, and the self-hosted AGENTS.md auto-Guru paragraph is updated through its owning seed path — seed fixed first, rendered surface regenerated, per the self-hosting rule (never hand-edit generated surfaces as a substitute).
5. **Live verification.** Post-fix, a spawned guru subagent and one factor subagent each successfully call at least one `code_*` tool (transcript-verified); the renderer's test suite (if `render_agent_surfaces` has tests) pins the allowlist content so a future template edit cannot silently drop the tools.
6. **Cross-host audit.** Other hosts' rendered role surfaces (Cursor rules, Codex skills, per `docs/agents/platform-mapping.md`) are audited for equivalent tool-restriction mechanisms; where a host has no allowlist concept, record N/A — audit-and-skip, no invented content.

## Scope

**Problem statement:** Rendered role-agent wrappers deny the MCP tools their own instructions (and the framework's delegation guidance) tell them to use, and the documented tool-inheritance premise is false for every agent that has an allowlist — making the framework's MCP-first posture unenforceable for role-typed subagents on every target repo.

**In scope:**

- Allowlist additions in the guru template (`render_agent_surfaces.py`) and factor-wrapper seed template; read-only constraint preserved.
- Premise corrections across seeds 180/100/050 + tree-wide grep for other instances; self-hosted AGENTS.md via its seed path.
- Re-render of this repo's surfaces; live subagent verification; renderer test pin; cross-host audit.

**Out of scope:**

- Any wording/posture guidance beyond the premise correction (owned by `1p9ql`).
- Granting `wave_*` lifecycle or write-capable tools to subagents (explicitly refused — reviewers stay read-only).
- Host-side deferred-tool loading behavior (ToolSearch mechanics) — host-owned; the allowlist is the framework's lever.
- New agent types or role docs.

## Risks

| ---- | ---------- |
| MCP server absent in a target repo/host session → wrappers name unattachable tools. | Naming an unavailable tool is inert in hosts' subagent semantics (tool simply absent); role docs' existing fallback guidance (grep when MCP absent) already covers behavior; verified in the live check on a no-MCP spawn if feasible. |
