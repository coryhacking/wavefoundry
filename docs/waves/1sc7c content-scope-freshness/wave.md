Title: Content Scope Freshness

## Objective

Fix the content-scope staleness cluster found during 1.12.0 release prep: content-scoped incremental builds stamp broad file hashes while embedding only their own content type, freezing the other type's index forever (the post-edit hook does this on every edit — incremental code indexing has effectively never worked on hook-enabled repos, and the documented `content=code` recovery is a no-op); `content=all` vs `content=code` build different code corpora; and dual-output code files (Python docstring chunks feed the docs table) go stale under BOTH scoped build types. When this wave closes, every table's change detection is coherent under any build scope, the automatic hook path keeps both semantic layers current, corpus membership is deterministic, and the fixes are proven by the poison-scenario fixture plus a live end-to-end edit→hook→search verification. Design-pass-gated: the mechanism choice (per-file meta-revert vs per-layer last-embedded hashes vs always-`all` automatic builds) is decided on census evidence before implementation.
