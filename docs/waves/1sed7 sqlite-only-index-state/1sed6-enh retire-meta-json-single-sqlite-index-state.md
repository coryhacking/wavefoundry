# Retire `meta.json`: One SQLite Authority for Semantic Index State

Change ID: `1sed6-enh retire-meta-json-single-sqlite-index-state`
Change Status: `implementing`
Owner: Engineering
Status: active
Last verified: 2026-07-12
Wave: `1sed7 sqlite-only-index-state`

## Rationale

Wave `1rsh9` made `.wavefoundry/index/index-state.sqlite` mandatory operational substrate for semantic-index bookkeeping, FTS5 retrieval, chunk registry state, freshness/attribution, secret-scan caching, and—through `1sc7c`—per-layer last-embedded hashes. The store already persists `build_file_meta` and `build_layer_meta`; the indexer then exports those rows back into `meta.json`. When the store write/export fails, however, the indexer silently writes a separately computed JSON payload and reports the build as successful (`indexer.py` ~4171–4185).

That fallback creates two operating modes and two state authorities. A JSON-success/SQLite-failure build can appear ready to `WaveIndex`, dashboard, health, and upgrade readers while the FTS registry, per-layer hashes, and other mandatory store residents are absent or stale. This is no longer a graceful degradation: so much shipped behavior depends on SQLite that publishing `meta.json` without a current store describes an index state the system cannot actually serve correctly.

The corrective architecture is one authority. Lance remains authoritative for chunk/vector existence; SQLite is authoritative for all semantic-index operational state. Missing, corrupt, or schema-incompatible SQLite is reset and rebuilt from repository/git/Lance under the existing derived-only contract. If that reconstruction cannot complete, the build fails visibly. `meta.json` is retired rather than retained as a manifest or fallback.

## Requirements

1. **One state authority:** `.wavefoundry/index/index-state.sqlite` is the sole semantic-index metadata/state authority. Remove the steady-state `meta.json` read, write, export, existence, freshness, and invalidation paths; no successful setup, build, update, health, search, dashboard, prune, or upgrade flow depends on `meta.json`.
2. **No fallback mode:** remove the `write_build_bookkeeping`/`export_meta_snapshot` failure fallback that writes `new_meta` directly. A store failure must route through the existing derived-only reset/recreate/reconcile path; if the store still cannot be made current, the index operation returns a structured failure and does not claim completion. Completion gates on explicit success from every mandatory resident—canonical build metadata, selected/all-layer state and hashes, chunk registry, FTS, and readiness-critical freshness state—rather than helpers swallowing errors; any optional resident is enumerated explicitly.
3. **Canonical build-state schema and epoch:** retain or refine the SQLite tables needed to replace every JSON field (`built_at`, model versions, per-layer chunker versions, walker version, available content, and per-file hash/stat/chunk-count bookkeeping). Add one typed build-state row with attempt ID, requested scope, constrained status, generation, start time, and completion time. A fresh/reset store starts `uninitialized`; a build transitions to `building`; a stale `building` epoch observed after its owning build/lock is gone is interpreted and reported as interrupted/dirty without inventing a separate authority; only a compare-and-set finalization for the same attempt ID may increment generation and mark `complete`. Schema changes follow the store's single sequenced version and reset/rebuild contract.
4. **Complete reader/writer census:** enumerate every production and test consumer of `meta.json` and record one disposition per site: SQLite query, Lance-derived value, repository-derived value, obsolete/removed compatibility path, or fixture migration. Extend the census to every semantic mutation entry point so none bypasses the build lock and epoch: normal/scoped builds, zero-change reap/heal, standalone FTS rebuild, optimize/compact/rewrite, setup, upgrade, and MCP/close-triggered paths. At minimum the census covers `indexer.py`, `index_state_store.py`, `server_impl.py` (`WaveIndex`, direct `code_lexical_response`, health, build/status resources), `dashboard_lib.py`, `dashboard_server.py`, `upgrade_wavefoundry.py`, `prune_framework.py`, setup/upgrade verification, and all fixtures.
5. **MCP reload/readiness semantics:** `WaveIndex` detects build completion and invalidates cached handles via SQLite build generation, not a JSON file signature. Every semantic/vector/FTS operation uses a seqlock-style epoch validation or a proven equivalent barrier: capture a complete `{attempt_id, generation}` token before the whole indexed operation, re-read it through a fresh short-lived query after Lance/FTS work, and return results only if the same token is still complete; otherwise discard and return a structured rebuilding/incomplete result. Dirty, uninitialized, interrupted, reset, malformed, missing, or unrecoverable SQLite state invalidates handles; an explicitly degraded live-filesystem lexical path may run, but Lance/FTS is never served as current in that state. Combined retrieval wraps the complete operation once rather than probing each backend separately, and no SQLite read transaction is held across Lance work. Uninterrupted indexed reads during in-place builds are not a requirement; preserving them would require a separately planned staged/double-buffered generation design.
6. **Dashboard, health, and performance parity:** dashboard health, `wave_index_health`, index-status resources, and build-status surfaces read SQLite/Lance directly and preserve or improve their current operator-visible fields. Store absence, structural failure, schema reset, reconciliation, and rebuild progress remain distinguishable. Preserve the dashboard's cached/event-driven collection model; use bounded SQLite aggregate and Lance row-count queries rather than materializing full path columns, and watch SQLite/WAL completion signals. Limit `FULL`-equivalent durability to the two small boundary commits of a mutating build; a proven true no-op performs neither. Record warm repeated probe/build measurements and structural probe-count/asymptotic budgets without brittle CI wall-clock gates.
7. **Upgrade and legacy convergence:** upgrade snapshots installed semantic-index versions from SQLite. An installation with legacy `meta.json` but no current store is treated as needing reconstruction/full convergence from repository/git/Lance; JSON is not imported as an alternate authority. After a successful compatible build, any stale `meta.json` is removed. The upgrade remains fail-safe when no prior index exists.
8. **Ordered consistency remains explicit:** a read-only preflight may prove a true no-op and leave generation unchanged. Before every semantic mutation—including nominal zero-change-path Lance reaping or registry/FTS healing—durably commit a SQLite `building` epoch with `FULL`-equivalent synchronous/fsync semantics so a process or power failure cannot leave a partially changed Lance index advertised by the prior completed generation. Readers fail closed while the current epoch is uninitialized, building, interrupted, reset, malformed, or otherwise incomplete. Lance writes then complete, reconciliation validates eligible paths, chunk/hash bookkeeping, and orphan removal, and one compare-and-set SQLite transaction for the same attempt ID publishes canonical build/per-layer metadata, advances completed generation, and marks the epoch complete. Cross-engine atomicity is not claimed. Recovery from a dirty epoch is bounded and visibly fails rather than looping indefinitely. A whole-store reset escalates any scoped request to mandatory all-layer repository-derived convergence, rebuilding registry/FTS/layer hashes and either recreating Lance or proving all reused Lance rows current before readiness.
9. **Architecture contract correction:** update the accepted SQLite ADR and search/index architecture docs so the derived-only reconstruction sources are repository/git/Lance, not `meta.json`; remove the "reader-compatible snapshot" contract and document failure semantics for an unrecoverable mandatory store.
10. **Regression coverage:** fixture-pin clean install, incremental build, content-scoped build, full rebuild, model/chunker/walker version transitions, long-lived MCP reload, dashboard/health reads, pre-store legacy upgrade, missing store, schema mismatch, structural corruption, SQLite write failure, interrupted Lance→SQLite sequencing, and stale-JSON cleanup on macOS/Linux/native-Windows-compatible paths.

## Scope

**Problem statement:** `meta.json` is a second semantic-index state surface whose fallback can report success while mandatory SQLite residents are unavailable or stale.

**In scope:**

- Semantic-index build bookkeeping and generation state in `index-state.sqlite`.
- Removal of `meta.json` production reads/writes and snapshot export.
- Migration of MCP, dashboard, health, upgrade, prune, setup, and test consumers.
- Legacy convergence by reconstruction, stale-file removal, architecture/spec/operator documentation, and fixtures.

**Out of scope:**

- Lance table schema or vector/chunk authority changes.
- Graph-store consolidation or refactoring `GraphStateStore`.
- New external database dependencies or non-stdlib SQLite extensions.
- Agent-memory/churn-decay behavior beyond preserving its existing store residents.
- Search ranking, chunking semantics, and FTS tokenizer changes.

## Decision Log


| Date | Decision | Reason | Alternatives |
| ---- | -------- | ------ | ------------ |
| 2026-07-12 | Create a dedicated wave for SQLite-only semantic-index state. | The work crosses store schema, indexer, MCP lifecycle, dashboard, upgrade, prune/setup, compatibility, and architecture contracts; it is materially broader than search freshness and has its own failure/recovery boundary. | **Fold into `1seav`:** rejected—bloats a release-focused search/freshness wave and creates overlapping acceptance criteria. **Fold into `1seax`:** rejected—ops hardening does not own semantic-index state architecture. |
| 2026-07-12 | Retire `meta.json` entirely; do not retain a thin manifest or fallback. | SQLite is already mandatory to correct operation. A JSON projection adds a second readiness signal and lets store failure masquerade as success; remaining consumers can query SQLite/Lance directly, including upgrade (which already reads graph SQLite pre-extract). | **Keep a compact manifest:** rejected—no load-bearing consumer requires a second file once migrated. **Keep JSON as recovery fallback:** rejected—creates a second authority and cannot reconstruct mandatory FTS/registry/per-layer residents. |
| 2026-07-12 | Legacy installations converge by reconstruction, not JSON import. | Repository/git/Lance are sufficient derived-state inputs; treating legacy JSON as authority would preserve the ambiguity being removed. | **One-time JSON import:** rejected unless implementation discovery proves a pre-store value cannot be reconstructed; any exception requires operator decision and an amended AC. |
| 2026-07-12 | Publish a durable SQLite `building` epoch before any Lance mutation and fail closed until a final verified completion transaction. | A completed generation records commit, not intent; without a pre-mutation marker a crash can leave partially changed Lance data behind an apparently valid old generation. Whole-store reset also erases untouched-layer version state, so scoped recovery must escalate to all-layer convergence. | **Generation advances last only:** rejected—cannot distinguish pre-commit crash from quiescent old generation. **Generation-version all tables/staged Lance snapshots:** deferred—unnecessary unless uninterrupted reads during builds become a requirement. |
| 2026-07-12 | Make the pre-Lance fence `FULL`-equivalent durable and finalize only by attempt-ID compare-and-set. | The store's ordinary `synchronous=NORMAL` policy may lose the latest commit on power failure; losing the fence recreates false readiness. CAS prevents a stale writer from publishing completion for a newer attempt. | **Use ordinary store durability:** rejected for the safety-critical fence. **Make every SQLite write synchronous FULL:** rejected as unnecessary overhead; target the fence/final publication contract and verify it. |
| 2026-07-12 | Fail indexed readers closed during in-place builds; do not promise uninterrupted indexed availability. | In-place Lance mutation cannot provide a coherent old generation once a build starts. Typed incomplete diagnostics plus an explicitly degraded live-filesystem lexical path are the smallest safe contract. | **Double-buffer immutable index generations:** valid future option if uninterrupted reads become a product requirement, but rejected here due additional storage, activation, retention, and cleanup complexity. |
| 2026-07-12 | Validate the same complete epoch before and after each indexed operation. | A pre-read alone is TOCTOU-unsafe because a writer can commit the fence and mutate Lance/FTS before results return. Two short queries close the result-publication race without holding SQLite/WAL state across Lance work. | **One pre-operation check:** rejected as unsafe. **Shared read lock/long SQLite transaction:** rejected due writer blocking/WAL pinning and cross-engine complexity. |

## Risks


| Risk | Mitigation |
| ---- | ---------- |
| Hidden `meta.json` consumer breaks after removal | Exhaustive production/test census plus zero-reference AC and live dashboard/upgrade smoke. |
| SQLite generation changes while WAL leaves the main file mtime unchanged | Query a committed generation row; never use SQLite file mtime as the invalidation authority. |
| Store failure turns formerly fail-soft builds into failures | Intentional: reset/rebuild first, then fail visibly rather than publish an unusable partial index; structured diagnostics and recovery instructions. |
| Legacy upgrade loses version-transition signal | Absence/current-store mismatch forces safe reconstruction/full convergence; fixture every supported predecessor shape. |
| Cross-engine crash leaves Lance and SQLite briefly divergent | Commit a dirty/building epoch before Lance mutation; fail readers closed; reconcile; publish metadata, generation, and complete state in one final SQLite transaction. |
| Scope overlaps `1seav` in `server_impl.py` | Resolved: operator ordered this wave first (2026-07-12); `1seav` reconciled to consume this wave's store API, so the overlap is a dependency, not a conflict. |
