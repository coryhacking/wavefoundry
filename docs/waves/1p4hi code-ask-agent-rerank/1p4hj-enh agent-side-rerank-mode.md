## Rationale

`code_ask`'s cross-encoder reranker (`BAAI/bge-reranker-base`, a local CPU model) adds **~30s of latency**, and gets worse under parallel queries because they serialize on the single shared local model (the "even longer in parallel" symptom). Yet `code_ask` already:

1. performs **NO LLM synthesis** — "the calling agent synthesizes from the returned citations" (`server_impl.py:15078`); and
2. has an RRF candidate path — the `reranked: false` field already returns Reciprocal-Rank-Fusion-ordered candidates when the cross-encoder is unavailable (`_rerank:836`, `_rrf_merge:851`).

So the cross-encoder is a redundant, slow, contention-prone middle step for the agent — its only consumer. The host LLM is a stronger reranker than `bge-reranker-base`, has the task context the reranker never sees, and parallelizes at the API. **Making agent-rerank the default** (the host agent is `code_ask`'s only consumer, and it always synthesizes) skips the cross-encoder and lets the agent rank+synthesize in the pass it already runs — eliminating the latency and parallel contention on every call. The cross-encoder is retained as the opt-in `"local"` mode (eval baseline / determinism / offline), and the default flip is **gated** on a quality check (AC-10) confirming agent-mode matches-or-beats it.

**The same logic extends past the cross-encoder to RRF itself.** Reciprocal-Rank-Fusion is *also* just a heuristic — it merges the per-index ranked lists into one order from rank positions, discarding score magnitude and the question's context. Cross-source fusion ("is this code hit or that doc hit more relevant to *this* question?") is exactly the contextual judgment the agent does better. So agent-mode skips **both** the cross-encoder *and* the RRF cross-source merge. Instead it returns **labeled per-index candidate lists** — each index's top-N, **budget-allocated across indexes** with a per-index floor so no modality is starved, **deduped for diversity** (near-duplicate chunks collapsed so the budget buys *distinct* evidence), as **full chunk `text`** (reranker-parity signal) — and lets the agent do the fusion, ranking, and synthesis. RRF and the cross-encoder become the same redundant machinery; the agent is the fuser-and-ranker.

## Requirements

1. `code_ask` accepts a `rerank` mode: **`"agent"` — the new DEFAULT** (skip the cross-encoder *and* the RRF cross-source merge; return labeled per-index candidates for the agent to fuse+rank+synthesize) — and **`"local"` — opt-in** (the current cross-encoder + RRF path, retained for the eval baseline, determinism, and offline use). The default flip is **gated on AC-10** (agent-mode must match-or-beat `"local"` on quality + recall before it ships as default).
2. In `"agent"` mode, `code_ask` invokes **neither** the cross-encoder (`_get_reranker` / `.rerank()`) **nor** the RRF cross-source merge as the final ordering — the agent fuses. (The per-index retrieval that feeds candidates still runs; what is skipped is the cross-encoder rescore and the RRF rank-fusion into a single order.)
3. **Candidate selection (agent-mode):** return **labeled per-index candidate lists**. Each candidate carries a `source` label (which index: code / docs / …). Within each index, take the index's own top-N (by its native score). Apply a **per-index coverage floor** — each contributing index always contributes its top-K (default K≈3), regardless of any cross-source comparison, so a single modality can never be starved. Then **fill the remaining token budget by relevance across all sources** (best-of-rest by score — the docs + code indexes share one encoding model, so cosine scores are directly comparable), tilted by `question_type` (navigational boosts code, brought forward from RRF). The floor is *only* for anti-starvation; the fill is **not** a forced per-source balance.
4. **Dedup for diversity:** collapse near-duplicate chunks (same symbol, overlapping line ranges, or near-identical text) within and across indexes *before* filling the budget — so the budget buys *distinct* evidence (different files/functions), not five chunks of one function. The dedup must not collapse legitimately-distinct chunks (different symbols that merely share a name). **When deduping across sources, PRESERVE the multi-source signal** — record that a candidate surfaced from more than one index (and its best per-source rank). RRF's value was not only ordering but *rewarding cross-source agreement* (a candidate strong in both code and docs is more relevant than one in a single list); dropping to a single `source` on collapse would discard that signal. The deduped candidate keeps the **set** of sources, so the agent can weigh agreement the way RRF did.
5. **Full chunk `text`, never vectors:** each candidate carries the **full chunk `text`** — the unit the cross-encoder reranks on (`:838`), NOT the 300-char `excerpt` truncation today's citation uses (`:14395`) — plus `{ref, path, lines, source, score, kind}`. Reranking the agent on *less* signal than the model it replaces would break the premise. **Never** embedding vectors (uninterpretable to the agent, token-wasteful; vectors stay server-internal). The chunk is a bounded index segment, so "full chunk" is not unbounded; the agent `code_read`s the winners for context *beyond* the chunk (escalation, not the default payload).
6. **Token/size budget:** the response is bounded by a token/size budget (mirror the `1p4es` `code_impact` cap — no unbounded blow-up). With per-candidate signal fixed at full-chunk parity, the levers are the **per-index counts + the cross-index allocation**, both derived from `budget ÷ chunk-size` — not a fixed gut number. Clamp budget/count ≤ 0; surface the total available per index.
7. The response is **self-describing** via a distinct `rerank_mode` field (`"agent"` | `"local"` | `"rrf_fallback"`). **Do NOT overload `reranked: false`** — today it means "degraded RRF fallback, model unavailable, lower quality", the *opposite* of a deliberate agent-fuse. The `answer`/`citations` contract is otherwise unchanged (citations gain a `source` label).
8. The `"local"` (opt-in) path is byte-for-byte the current cross-encoder + RRF behavior — the eval baseline and any determinism-sensitive caller can pin it.
9. Docs (the `code_ask` docstring + `docs/specs/mcp-tool-surface.md`) document the mode, the "agent fuses" model, the budget tradeoff, the `source` label, and when to prefer each.

## Scope

**Problem statement:** the cross-encoder rerank is a ~30s latency tax and a shared-local-model serialization bottleneck, and RRF cross-source fusion is a context-blind rank heuristic — both are redundant middle steps for the agent, which already synthesizes and out-judges them.

**In scope:**

- A `rerank` mode parameter on `code_ask` (`"agent"` default | `"local"` opt-in), with validation.
- The agent-mode candidate pipeline: per-index top-N + per-index floor + cross-index budget allocation + dedup + full-chunk + `source` labels; **no cross-encoder, no RRF cross-source merge**.
- A self-describing `rerank_mode` field + the `source` label on citations.
- Retain the cross-encoder + RRF path under `"local"`.
- Docstring + `mcp-tool-surface.md` updates.
- A quality + recall check (AC-10) gating the default flip.

**Out of scope:**

- **Removing** the cross-encoder or RRF — both stay under `"local"` (offline-capable design, eval baseline, determinism).
- Graph/structural signal in candidate selection — that is `1p4hu-enh graph-signal-retrieval-fusion` (same wave).
- Score-aware (magnitude-normalized) cross-source fusion — a possible later refinement; agent-fuses supersedes it here.
- Applying the mode to `code_search` / `docs_search` — candidate follow-up.
- Calling an external LLM **from the MCP server** — the server stays offline-capable; the agent-fuse happens in the *already-present* host agent.

- [x] AC-3: Each candidate carries the **full chunk `text`** (reranker parity, `:838` — NOT the 300-char `excerpt`, `:14395`) plus a `source` index label, and **never** an embedding vector. Verified by tests asserting full-chunk-text payload, the `source` label presence, and the no-vector invariant.
- [x] AC-4: **Per-index floor (anti-starvation) + relevance fill, count governed by a text BUDGET (not a fixed N)** — agent-mode includes each contributing index's top-K (default K≈3) regardless of cross-source ranking, then fills by **relevance across all sources** (best-of-rest by score) — **NOT a forced per-source balance** (no better than concatenating `code_search` + `docs_search`). The returned **count is an outcome of three limits, never a fixed N**: (1) the **relevance drop-off** (`AGENT_RELEVANCE_DROPOFF`=0.85×top weighted score) — stop past the top cluster (*enough to answer, no noise*); (2) the **text budget** (`AGENT_TEXT_BUDGET_CHARS`=16000 ≈ 4k tokens, set by the AC-10 sweep) of cumulative full-chunk text — the *don't-over-feed-the-agent* ceiling, so the count adapts to chunk SIZE (many small chunks fit, few large ones — packed greedily by relevance, the floor counted against it); (3) a loose **count backstop** (`AGENT_CANDIDATE_MAX`=40) for pathological tiny-chunk cases. Cross-source scores are comparable (docs + code share `bge-small-en-v1.5`); the **navigational tilt is brought forward from RRF** (code ×`RRF_NAVIGATIONAL_CODE_WEIGHT`=1.5), and the **explanatory doc-demotion is applied PRE-selection** (validated prior: code > narrative docs) so the demoted narrative tail falls below the drop-off and is trimmed, not selected-on-raw-cosine then padded in. Verified by: starvation, relevance-fill, navigational-weight, drop-off, and **budget-by-chunk-size** tests (same budget → many small chunks vs few large). Real-index behavior: the budget binds for content-rich queries (count adapts to chunk size — **22 / 27 / 32 / 37** at the 24000 sweep setting, proportionally fewer at the chosen 16000), the drop-off binds when relevant content runs out (e.g. 13, well under budget), and off-topic queries return at a low band (a relative cutoff can't detect a weak top — the agent reads the low scores + `gaps`; an absolute floor is too brittle on bge's compressed scale). **Budget/cutoff calibrated by AC-10** (recall-vs-context-size sweep): budget set to **16000** (recall saturates ≤8000c; 24000 was over-provisioned), drop-off 0.85 — operator to run cross-project synthesis validation before ship.
- [x] AC-5: **Dedup / diversity** — near-duplicate chunks (same symbol / overlapping lines / near-identical text) collapse so the returned set is distinct; legitimately-distinct chunks are NOT collapsed; **a cross-source duplicate retains a multi-source marker** (the set of sources + best per-source rank) so the cross-source-agreement signal RRF provided is not lost. Verified by: a duplicate-chunk test (one survives); a distinct-same-name test (both survive); and a cross-source test (a candidate in two indexes survives once with both sources recorded).
- [x] AC-6: **Token/size budget** bounds the response (mirror `1p4es` cap — no blow-up); counts derive from `budget ÷ chunk-size`; degenerate inputs handled (0 results, budget/count ≤ 0 clamped). Verified by a size-bound + degenerate-input test.
