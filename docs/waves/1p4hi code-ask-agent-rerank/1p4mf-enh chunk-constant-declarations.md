## Rationale

`code_ask` cannot answer "what value/model/flag is X" when X is a module-level constant, because **the chunker never emits constant declarations as chunks**. `chunk_python` (chunker.py:456-500) emits chunks ONLY for `FunctionDef`/`AsyncFunctionDef`/`ClassDef` + docstrings; the `else` branch recurses without emitting, so `RERANKER_MODEL = "BAAI/bge-reranker-base"` (indexer.py:74, module-level) lives in **zero chunks** — verified: the motivating query returns function *bodies* that read the constant plus a test holding the value literal, never the declaration.

This is the **foundational** fix: make the declaration (with its value) a retrievable chunk, so `code_ask`'s semantic pass can return it. It is the missing substrate the sibling changes assumed — in particular it **revives `1p4lr`** (the candidate-side declaration boost, currently blocked because no constant chunk exists to fire on). A bare `NAME = value` chunk is short and context-poor — the exact rank-11 shape `bge-small` under-ranks — so the chunk makes the constant *retrievable* and `1p4lr`'s boost makes it *rank*; they ship coupled. The structural complement (`1p4ls` graph constant nodes) answers the orthogonal "who reads X / blast-radius" questions and stays.

There is exactly one precedent already in the tree: `chunk_ts` (chunker.py:2180-2219) chunks top-level `export const` declarations one-per-decl. This change **generalizes that single case** to Python (AST) and the other core languages.

## Requirements

1. **One chunk per module-level constant.** `chunk_python` emits a chunk for each top-level `ast.Assign`/`ast.AnnAssign` whose target is an `ast.Name` (mirroring the `chunk_ts` export-const shape): `id = {path}::{CONST_NAME}`, `kind = "code"`, section breadcrumb `{stem} > {CONST_NAME}`, span = the declaration line(s) **plus any contiguous leading comment** (the Go `comment_blocks` / JS `doc_ends` pattern). **Prefix the embedded chunk TEXT with the breadcrumb** (`{stem} > {NAME}\n\n{decl}`, as the chunkers at chunker.py:1805/2005 already do — the `chunk_ts` export-const precedent at :2216 emits BARE text and is the rank-11 shape to avoid): injecting the symbol name into the embedding raises bare-cosine retrievability *for free* and may **decouple the bump from `1p4lr`'s boost** (AC-6 measures bare vs breadcrumbed vs boosted). For **class-body** constants the id is `{path}::{Type}.{NAME}` (a disambiguator vs a same-named member — guard for id-uniqueness). No `Chunk` dataclass change.
2. **Scoping gate (guard #1) — per-language constant MECHANISM + scope, NOT a uniform `UPPER_SNAKE` filter.** Detect constants by each language's real mechanism (keyword / structure / grammar node) and **scope-gate to module / file / namespace / type top level** — never function/block/comprehension locals. **Scope is the load-bearing discriminator** (in Swift/Kotlin/Rust/PHP/Python a local is the SAME node/AST type as a constant, so the ancestor/parent-chain gate, not casing, is what separates them). **Gate on CASING only for Python** (`UPPER_SNAKE` `^[A-Z][A-Z0-9_]*$`, len ≥2-or-has-`_`/digit, with `typing.Final` as a casing-independent override); **Ruby** uses the tree-sitter `constant` node type (capital-initial, grammar-encoded — test the node type, don't re-grep). For **every keyword language (JS, TS, Swift, Java, Kotlin, C#, Go, Rust, PHP) do NOT gate on casing** — idiomatic constants are camelCase/PascalCase/MixedCaps (Swift `apiURL`, C# `MaxRetries`, Go `StatusOK`, JS `defaultConfig`), so an `UPPER_SNAKE` filter silently drops the majority. Include **private/unexported** constants (the existing `export const`-only path is a bug to close); iterate **per-name** for multi-declarator statements (`const A=1,B=2`, `A=B=5`); **enum members** are distinct nodes for **TS + Swift only** (excluded for the other nine — already inside their type container). The exact per-language rule is the **Per-Language Constant Detection** table below. (Casing may be recorded as a ranking-only `is_screaming_snake` flag, never an include filter. The "≥2 tokens" requirement is a `1p4lr` boost-key concern, NOT a chunk-inclusion gate — it does not apply here.)
3. **Merge guard (guard #2 — load-bearing).** `CHUNK_MIN_LINES = 2` (chunker.py:230) + `_merge_small_chunks` (chunker.py:1665-1683) SILENTLY fold a 1-line constant chunk into the preceding chunk — **NOT just "after a function": ANY 1-line constant following ANY prior chunk (including another constant) is folded, and the survivor keeps the PREDECESSOR's id** (verified live: adjacent `A = 1`, `B = 2` collapse into one chunk `id=…::A`, text `A = 1\nB = 2` — `B`'s id never exists, so `code_ask` can't cite `B` and `1p4lr` can't boost it; this hits the *common* consecutive-constants case, not an edge case). Tag constant chunks with a **specific section suffix** (state the exact string) excluded from the merge — add the exclusion in BOTH the `is_imports`-style predicate (chunker.py:1666) AND the `last_code_idx` target-finder (chunker.py:1652) — and require **every language path** emit the identical suffix. Without this guard, consecutive constants collapse into one mis-id'd chunk and the fix **fails invisibly**.
4. **`CHUNKER_VERSION` 25 → 26 in the same change.** Chunk-shape change → consumer indexes must re-extract; the indexer auto-escalates a chunker-version mismatch to a FULL code-corpus rebuild (indexer.py:1990-2019), and the upgrade flow routes every consumer to `phase_index_rebuild`. This is the **heaviest lever in the pipeline** — strictly heavier than `1p4ls`'s graph-only rebuild.
5. **All core supported languages — STAGED behind one bump (the local eval can only gate what has local source).** **Python (stdlib AST) is REQUIRED and locally gated** — the one tree-sitter-independent path and the precondition for shipping the `CHUNKER_VERSION` bump (AC-6 runs on it). **JS/TS** lands with it (existing `export const`, extended to non-exported `const` — close the `_JS_EXPORT_CONST_RE` export-only bug at chunker.py:2085). The other **eight languages (Go, Java, Kotlin, C#, Rust, Swift, Ruby, PHP) are ADDITIVE passes under the same bump** — this repo has **87 `.py` / 1 `.js` / 0 other-language** source, so they **cannot be recall-gated locally**; their fixtures land with the bump and their per-language bloat/recall is a **named post-bump checkpoint measured against the p4ea multi-lang test-pack** (the gating oracle for source-absent languages), NOT silently skipped by the local gate. Adding a language later is additive (same chunk shape) and does NOT require a second `CHUNKER_VERSION` bump.
6. **Gate the version bump on a local eval (guard #3).** Build `CHUNKER_VERSION=26` LOCALLY and run the AC-10 known-answer recall harness BEFORE the bump reaches any consumer. The bump ships only if constant-chunk (+ `1p4lr` boost) lifts the "what value is X" class **without a dilution regression** on the non-constant known-answer queries. If it cannot beat the `1p4ls`-only structural path, prefer the grouped-per-file fallback or do not bump.

7. **Reconcile the four constant-detection surfaces.** After `1p4mf`/`1p4ls`, FOUR constant detectors exist: the `1p4mf` chunk lane, the `1p4ls` graph lane, and the **pre-existing** `code_outline` (UPPER-only Python gate, server_impl.py:~9596) + `code_constants` (column-0 Go, ~10100) tools. They will silently disagree (a user finds `MaxRetries` via `code_ask` but `code_outline` shows nothing on the same file — visible incoherence). Either factor per-language detection into **one shared helper** consumed by `chunker`, `graph_indexer`, and the two tool surfaces, OR explicitly **scope the two legacy tools out as known-divergent** (document the UPPER-only / column-0 behavior, file a follow-up, add a Journal Watchpoint). Decide + document — do not leave four detectors silently diverging.

Document the delta so the chunk / graph / boost surfaces stay coherent.

**Grammar-availability caveat (silent-failure surface):** Ruby/Kotlin/PHP pass `regex_fn=None`, so a missing tree-sitter grammar drops straight to line-windows with ZERO const awareness; Python falls to `chunk_line_window` on `SyntaxError`. **Verify the grammars are installed in `~/.wavefoundry/venv`** ([[venv-for-test-agents]]) before claiming a per-language gate works — else the gate is vacuous. Do NOT add a regex fallback that fakes const detection from broken text.

## Scope

**Problem statement:** Module-level constants are absent from the chunk corpus, so `code_ask` cannot retrieve a constant's value semantically — the foundational gap behind the `1p4hj` AC-10 `RERANKER_MODEL` miss.

**In scope:**

- `chunker.py`: per-language module-level constant chunk emission (Python AST first; Go/Java/Kotlin/C#/Rust/Swift/Ruby/PHP staged regex/tree-sitter passes; JS/TS already present), the scoping gate, the `_merge_small_chunks` exclusion, and `CHUNKER_VERSION` 25 → 26.
- The local-eval gate (AC-10 harness on a v26 build) before the bump is considered shippable.
- Bloat measurement (Python here; other languages against p4ea) recorded.

**Out of scope:**

- Local/function-scope constants, type aliases, enums, struct fields (named module-level constants only — a bounded, principled boundary).
- The candidate-side rank boost (`1p4lr`) and the structural graph constant nodes (`1p4ls`) — separate changes; this one only makes the constant a chunk.
- A grouped per-file "module constants" chunk shape (kept as the documented fallback if the per-constant eval gate fails).
