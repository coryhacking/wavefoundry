Title: Restructure install flow into two phases backed by a checkbox install log and a validating audit tool — ship 1.5.0

## Objective

Downstream consumer install retrospective surfaced a cluster of related defects: the install completed without producing required artifacts (per-role docs, per-role journals, public prompts for seeds 175/176, second/third hook in settings.json); the verification checklist didn't catch the gaps; the agent ran out of MCP coverage on roughly half the seed catalog mid-install; the dashboard rendered silent empty panels instead of recovery actions; and the entire install assumed MCP availability from the start, even though MCP isn't reachable until the framework is installed. This wave restructures install into two phases with a markdown-native state machine (a pre-populated checkbox install log that ships in the zip) and adds a single MCP tool (`wave_install_audit`) that gates advancement on docs-lint + artifact-presence checks. The log makes Phase 1 MCP-free (agent reads file, ticks box, advances) and Phase 2 MCP-validated. Closes the root cause across all six clustered defect groups. Ships as part of **1.5.0** jointly with a follow-on wave admitting `1p397` (chunker oversized-chunk guard) and `1p399` (indexer drift detection); both waves close before the 1.5.0 release tag so consumers absorb the install-flow restructure AND the chunker fix in the same upgrade rather than across two version bumps.

## Journal Watchpoints

- **The install log IS the install state machine, not a side artifact.** Every bootstrap step writes a `[x]` when complete. Every `wave_install_audit` call validates the checked rows have produced their expected artifacts. Drift between log state and on-disk state is the failure this wave exists to prevent — if any change introduces a log-write path that doesn't have a corresponding artifact-verify path, the verification gate is broken by construction.
- **Phase 1 has NO MCP dependency.** Bare shell + python + filesystem only. If any C1 step requires `wave_*` MCP tool calls, the bootstrap is broken (MCP isn't running yet — that's literally what Phase 1 is installing). Phase 2 may use MCP freely.
- **`wave_install_audit` runs docs-lint on every call.** Lint failures block advancement. This is the lint-as-you-go discipline that catches the metadata-block and manifest-schema gaps from the retrospective immediately, not at some later wave_audit. Implementation must not skip lint when artifact-verification fails — both checks run, both surface as diagnostics.
- **`install-wavefoundry.md` (renamed from INSTALL.md) is the agent-readable bootstrap entry.** Filename matches the shortcut phrase. Content tells the agent: "you've extracted the zip; the install log is at `wavefoundry-install-log.md`; start at the first unchecked row." This is the load-bearing entry point inside the zip — keep it discoverable.
- **Pre-extraction discoverability is a separate problem from in-zip artifacts.** C6 handles the "zip dropped at repo root → agent doesn't know what to do" gap via the GitHub repo README + Releases page description. No zip-internal file can solve that case because the zip hasn't been extracted yet. Don't conflate the two surfaces in implementation.
- **Removing the pycache hook is the right architecture, not a regression.** The hook was treating a symptom (`__pycache__` present on disk) when the real bug was docs-lint flagging gitignored content. C5's removal of the third hook is paired with the docs-lint exclusion — they ship together, neither alone.
- **The three councils MUST always be surfaced in every install.** red-team, wave-council, and archetype-council role docs under `docs/agents/specialists/` are framework-default review surfaces, not optional. seed-012 (Phase 2 row 2.4) and seed-050 instruct the agent to read seeds 225, 215, 236 in full when generating these role docs — generating thin generic templates misses the framework's intent and produces shallower content than the authoritative seeds carry. C4's docs-lint `Role:` field enforcement is what makes missing role docs immediately visible; C3's seed-050 authoritative-pointer block (which gains seed-236 in addition to the originally-named 214/215/216/221-225) is what makes the rich content reachable.
- **Archetype Council scope is general-purpose, not text-only.** The framework's own `docs/agents/specialists/archetype-council.md` (after wave 1p33j) names the broader scope: plans, design docs, code, prose, decision narratives, naming, AC formulation — any artifact where orthogonal stance-based lenses are what the work rewards. Downstream installs that generated a thinner "text-only" framing missed this. Implementations of seed-050 must preserve the broader scope from seed-236 when generating the role doc.
- **Six changes is a lot for one wave.** The 1p337 prepare-council warned about late-admit. This wave admits all six up-front (no late-admit), but the strongest-challenge from prepare-council is likely to be "should this be 2–3 smaller waves instead." Be prepared with the cluster-coherence answer: these are not independent fixes; they share a root cause (install verification can pass while the install is incomplete) and the install-log mechanism is the root-cause fix. Splitting would create a partial-fix release that ships verification gaps with the verification gaps' diagnostic but no remediation surface.
