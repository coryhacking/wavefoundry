# Two-Phase Install With Checkbox Log And Entry Doc

## Rationale

The downstream consumer install retrospective showed the install completing with missing artifacts (per-role docs, per-role journals, public prompts for seeds 175/176, second/third settings.json hook) and no verification surfacing the gap. The single-phase seed-010 is 378 lines covering ~50 outputs in a 15-step sequence; there's no minimum-viable subset, no audit trail of what ran successfully, no recovery point if the agent stalls mid-install, and the entire flow assumes MCP availability even though MCP isn't reachable until the framework is installed.

This change restructures install around a **markdown-native state machine**: a pre-populated checkbox install log (`wavefoundry-install-log.md`) that ships in the zip at root next to a renamed agent-readable entry doc (`install-wavefoundry.md`). The install is split into Phase 1 (harness setup — no MCP dependency) and Phase 2 (project discovery — MCP available after restart). Each row in the log is a checkbox `[ ]` that points at a seed prompt; the agent reads the file, finds the first unchecked row, executes the referenced seed, marks `[x]`, advances. The log gives the install:

1. A discoverable entry point at the zip root (`install-wavefoundry.md` — filename matches the `Install Wavefoundry` shortcut phrase, so an agent dropped into a fresh extraction can find it via filename pattern).
2. A persistent state machine the agent can't lose track of (markdown checkboxes survive crashes, restarts, and agent swaps).
3. An audit trail (the log IS the trail — `[x]` rows are what ran).
4. A natural restart boundary (the row that ends Phase 1 says "STOP — instruct operator to restart agent for MCP availability").
5. A consumable contract for `wave_install_audit` (companion change `1p35h`) — the tool reads the log and validates.

## Requirements

1. **`wavefoundry-install-log.md` template defined and shipped in the zip at root.** Pre-populated with all bootstrap steps for both phases. Universal template (every install starts with the same rows); the agent checks-as-not-applicable when a row doesn't fit the project rather than the template being conditional.
2. **Each log row points at a seed prompt.** Format: `- [x] N.M — seed-NNN-name (one-line intent + expected artifact path)`. Rows are not free-form prose; they are seed pointers + artifact pointers. The agent reads the seed, executes it, and the artifact path tells the verification step what to check.
3. **`install-wavefoundry.md` at zip root** (renamed from `INSTALL.md`). Agent-readable bootstrap entry. Says: you've extracted the zip; macOS hides `.wavefoundry/` (Cmd+Shift+. to show); the install log is at `wavefoundry-install-log.md`; start at the first unchecked row; if you're done with Phase 1 the next instruction is to restart your agent so MCP becomes available.
4. **`INSTALL.md` is removed** (replaced by `install-wavefoundry.md` at the same path). No backward-compat alias; consumers downloading 1.5.0 see the new filename. Older releases retain the old filename.
5. **Seed-010 split into seed-011 (Phase 1) and seed-012 (Phase 2).** Original `seed-010` becomes a thin router/index that points at the two phase seeds and explains the split for operators reading the seed-NNN catalog directly. Each phase seed is focused on its own steps without the full 378-line load.
6. **Phase 1 has NO MCP dependency.** Every Phase 1 step runs via shell, Python, or filesystem edits only. The shortcut phrase invocation surface is sufficient (`Install Wavefoundry` brings the agent to the entry doc; the entry doc + log carry it the rest of the way without MCP).
7. **Phase 1 includes lifecycle epoch set as step 1.1.** Resolves the chicken-and-egg ordering bug from the retrospective: epoch is set in `workflow-config.json` before any lifecycle ID is generated. All subsequent IDs use the correct epoch.
8. **Phase 1 ends with an explicit restart marker.** Last unchecked Phase 1 row reads `[ ] 1.N — STOP: Instruct operator to restart agent so MCP becomes available before Phase 2`. The agent marks this `[x]` only after instructing the operator.
9. **Phase 2 first row is `wave_install_audit` invocation.** Confirms Phase 1 artifacts present before any Phase 2 work begins. If Phase 1 left gaps, Phase 2 doesn't start.
10. **`build_pack.py` ships the install-log template + the entry doc.** Both files live at the zip root (outside `.wavefoundry/`). Template content is generated by build_pack so it's always current with the seed catalog and shortcut phrases.
11. **The template is generated from a single source-of-truth file in the framework**, not hand-edited in build_pack. Source: `.wavefoundry/framework/templates/wavefoundry-install-log.md.template`. build_pack reads it and embeds it in the zip. Future seed additions update the template; build_pack picks them up.
12. **Existing in-repo INSTALL.md is removed from this self-host.** The self-host shouldn't carry the old filename when this change ships.

## Scope

**Problem statement:** Single-phase seed-010 with no state machine + no audit trail + MCP-dependency assumption produces installs that complete without verifying their own outputs. The verification gap is the root cause across at least four distinct defect groups in the consumer retrospective (missing role docs, missing journals, missing public prompts, missing hooks).

**In scope:**

- Split seed-010 into seed-011 (Phase 1) and seed-012 (Phase 2); make seed-010 a router
- Define `wavefoundry-install-log.md` schema + template source file
- Rename + repurpose `INSTALL.md` → `install-wavefoundry.md` at zip root
- Wire build_pack to ship both files
- Move lifecycle epoch set to step 1.1 (epoch-before-IDs ordering fix)
- Phase 1 / Phase 2 boundary marker (restart-agent step)
- Phase 2 first step is `wave_install_audit` call

**Out of scope:**

- The `wave_install_audit` MCP tool itself (companion change `1p35h`)
- Seed-content polish of seeds referenced by the log (separate concerns; covered by `1p35j`, `1p35l`)
- Pre-extraction discoverability (covered by `1p35p` for the GitHub repo README + Releases page)
- Index coverage gaps for seed retrieval (covered by `1p35j`)
- Per-role doc/journal generation correctness (covered by `1p35l`)

## Risks

| Risk | Mitigation |
|---|---|
| Universal template includes rows for personas/specialists that don't fit a given project | The agent marks not-applicable rows `[~]` (existing AC marker for "intentionally not met" from wave 1p32k); `wave_install_audit` accepts `[~]` as terminal, not pending. |
| Template drift: future seed additions don't update the template | The template is the source-of-truth file; adding a seed without adding a row is caught by manual review. Could add a docs-lint check that every public seed has a corresponding log row, but that's scope-creep for this change. |
| Existing operator workflows assume `INSTALL.md` filename | The new release is 1.5.0; consumers upgrading see the renamed file. Older releases retain the old filename in their zips. No silent break of in-flight installs (the file only exists post-install, post-extraction). |
| Phase 1 step adds an MCP dependency by accident in a future change | Journal Watchpoint names this explicitly. AC-13 grep-tests for `wave_*` references in seed-011. Council review catches it. |
| Build_pack changes accidentally drop one of the two files from the zip | AC-15 test_build_pack tests verify both files present. CI dogfood verifies on every release. |
