## Rationale

`12tm5-enh python-tool-venv-bootstrap` moved framework dependency installation from `sys.executable` into `~/.wavefoundry/venv`, but all shell launchers and MCP server configs still invoke `python3` directly. On a fresh install this breaks immediately: the venv is bootstrapped by `setup_index.py`, but every other entry point — `docs-lint`, `docs-gardener`, `upgrade-wavefoundry`, `wave_dashboard`, `register-codex-mcp`, and the MCP server itself — still calls the system Python which has no framework packages.

The canonical source for all generated surfaces is `render_platform_surfaces.py`. Updating it and re-running it propagates the fix to all downstream surfaces atomically. A remaining consistency pass is also required for non-generated internal subprocess paths and index-time dependency installation so the repository stops depending on `sys.executable` once the shared venv exists.

## Requirements

1. Shell bin launchers (`docs-lint`, `docs-gardener`, `upgrade-wavefoundry`, `wave_dashboard`) resolve the venv Python via `${WAVEFOUNDRY_TOOL_VENV:-$HOME/.wavefoundry/venv}/bin/python` and fall back to `python3` only if the venv Python binary is absent (not yet bootstrapped).
2. `register-codex-mcp` passes the venv Python as the MCP server interpreter to the `codex mcp add` command, with the same fallback logic.
3. MCP server JSON configs (`.mcp.json`, `.cursor/mcp.json`, Junie, Air/other copy-ready entry) use the resolved absolute venv Python path at render time; `render_platform_surfaces.py` expands `~/.wavefoundry/venv/bin/python` to an absolute path when generating these configs.
4. Generated Windows host-hook launchers (`.cmd`) use `%WAVEFOUNDRY_TOOL_VENV%` with fallback to `%USERPROFILE%\.wavefoundry\venv\Scripts\python.exe`. Operator-facing bootstrap and upgrade wrappers remain POSIX/WSL2-oriented and do not promise native Windows parity.
5. `render_platform_surfaces.py` is re-run after the template changes so all generated files on disk reflect the new interpreter.
6. The `gate` script (`#!/usr/bin/env python3`) is exempt — it has no framework package dependencies.
7. Direct internal indexing subprocesses and index-time dependency installation use the shared tool venv rather than `sys.executable`, except for the venv bootstrap step itself.
8. Architecture/runtime docs describing the Python environment reflect the shipped `~/.wavefoundry/venv` contract rather than the superseded pre-implementation plan.
9. Operator-facing runtime subprocesses that may import framework dependencies or should remain on the same managed runtime (`upgrade_wavefoundry.py`, `server_impl.py` dashboard/upgrade/index helpers) resolve the shared tool venv explicitly instead of relying on inherited `sys.executable`, with bootstrap-only exceptions documented.
10. Generated Python hook chains and hook simulators resolve the shared tool venv when delegating from one Python hook entrypoint to another, so wrapper-level venv selection is not lost during intra-hook subprocess hops.
11. Operators get a clearer bootstrap entrypoint: `setup_wavefoundry.py` delegates to `setup_index.py`, exposes the same CLI surface, and becomes the preferred documented command without breaking existing `setup_index.py` call sites.
12. `.wavefoundry/bin/setup-wavefoundry` exists as the canonical shell launcher for the new bootstrap entrypoint and is generated by `render_platform_surfaces.py` alongside the other operator wrappers.

## Scope

**In scope:**
- `render_platform_surfaces.py` — update all bin launcher templates, MCP JSON config generation, and Windows bat templates
- Re-run `render_platform_surfaces.py` to regenerate all surfaces on disk
- `docs/prompts/install-wavefoundry.prompt.md` and `docs/prompts/upgrade-wavefoundry.prompt.md` — update any python3 references in MCP registration steps
- `indexer.py`, `dashboard_server.py`, and `server_impl.py` — align direct indexing subprocesses and dependency installation with the shared venv
- `upgrade_wavefoundry.py` and `server_impl.py` — resolve the shared tool venv explicitly for operator/runtime subprocesses instead of inheriting `sys.executable`
- generated host hook chains in `render_platform_surfaces.py` — keep Python-to-Python hook delegation on the shared tool venv
- bootstrap operator surface — add a clearer `setup_wavefoundry.py` alias for environment/bootstrap + index setup
- bin launcher surface — add `.wavefoundry/bin/setup-wavefoundry` via `render_platform_surfaces.py`
- `docs/architecture/current-state.md` — remove the stale pre-implementation Python environment note

**Out of scope:**
- `gate` script — stdlib-only, no venv packages needed
- `run_tests.py` — development tool, always run directly
