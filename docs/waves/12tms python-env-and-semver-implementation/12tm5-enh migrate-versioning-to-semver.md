## Rationale

Wavefoundry uses `YYYY-MM-DDx` date strings as its release identifier in `VERSION`, `framework_revision`, artifact filenames, and upgrade ordering. That format is a weak public contract: it does not communicate compatibility intent, upgrade scope, or the difference between patch, minor, and major changes. Before the first public release, Wavefoundry will adopt `MAJOR.MINOR.PATCH` semver. The upgrade ordering comparison (`compare_versions()`) relies on lexicographic string ordering which works for date strings but breaks for semver (`"1.10.0" < "1.9.0"` lexicographically). All version comparison must be replaced with proper semver parsing. A migration path is required for repos already carrying date-based `framework_revision` values.

The migration is a two-phase release:

- **v0.9.0 (bridge release):** First semver-format pack internally. It includes the new `check_version.py` with `_to_version()`, but the artifact keeps the old date-style zip name (`wavefoundry-YYYY-MM-DDx.zip`) so legacy pre-semver upgrade flows can adopt it directly. Once landed, future semver-named packs upgrade normally.
- **v1.0.0 (first clean semver release):** Full new packaging workflow. Projects already on v0.9.0 upgrade automatically; the new `check_version.py` handles the comparison correctly.

A shared release directory at `~/.wavefoundry/dist/` provides a single drop location for built zips. The packager writes there by default and the upgrader searches it automatically, picking the highest semver zip.

## Requirements

1. Replace `YYYY-MM-DDx` date strings with `MAJOR.MINOR.PATCH+<build>` semver in `VERSION` and `framework_revision`. The build suffix is generated by `lifecycle_id.py --prefix-only` (e.g., `12tm5`), producing versions like `0.9.0+12tm5`. Zip artifact filenames use a `.` separator instead of `+` for filesystem safety from `1.0.0` onward; the one-time `0.9.0` bridge release keeps the old date-style zip name for legacy adoption.
2. Replace lexicographic string comparison in `compare_versions()` with `packaging.version.Version`-based comparison; strip build metadata before comparing precedence (build metadata is ignored for ordering in semver).
3. Handle mixed-version upgrades: repos with date-based `framework_revision` are treated as pre-`0.9.0` (mapped to `"0.0.0"` for comparison purposes).
4. Update upgrade hooks that use raw string comparison (`ctx.from_version < "2026-05-19a"`) to use `packaging.version.Version` with a date-string fallback.
5. Update `build_pack.py` to accept a semver base version (`--version 0.9.0`) and auto-generate the build suffix via `lifecycle_id.py --prefix-only`; default output directory is `~/.wavefoundry/dist/` (created if absent).
6. Update `upgrade_wavefoundry.py` to search `~/.wavefoundry/dist/` for `wavefoundry-*.zip` files when no zip path is given, select the highest version zip (sort by MAJOR.MINOR.PATCH descending, then build prefix lexicographically descending for ties), and fall back to the current project directory if the distribution folder is empty or absent. Filenames that do not match the `wavefoundry-MAJOR.MINOR.PATCH.<build>.zip` pattern are silently skipped during discovery.
7. Declare `packaging` as a runtime dependency in `pyproject.toml` (coordinate with `12tm5-enh python-tool-venv-bootstrap` which creates that file).
8. For the v0.9.0 bridge release, `build_pack.py` keeps the old date-style zip artifact name (`wavefoundry-YYYY-MM-DDx.zip`) while stamping `VERSION` and `framework_revision` to `0.9.0+<build>`. This lets legacy pre-semver zip-adoption flows land the bridge pack directly; subsequent releases then use semver artifact naming.
9. `_to_version()` must use an explicit date-string regex (e.g., `\d{4}-\d{2}-\d{2}[a-z]`) rather than catch-all logic; unknown formats that are neither valid semver nor date-shaped should raise a clear error, not silently map to `"0.0.0"`.
10. Update tests (`test_build_pack.py`, `test_upgrade_wavefoundry.py`) and operator docs to reflect the new versioning contract, the v0.9.0 bridge migration path, and the `~/.wavefoundry/dist/` workflow.

## Scope

**Problem statement:** The date-based version format is embedded across packaging, upgrade, and docs surfaces and is not a strong public release contract. Version comparison code relies on lexicographic ordering that breaks for semver. Existing projects run the old upgrade script, which cannot compare semver against date strings correctly.

**In scope:**

- `build_pack.py`: accept `--version MAJOR.MINOR.PATCH`; stamp `VERSION` and `framework_revision` as `MAJOR.MINOR.PATCH+<build>`; name zip as `wavefoundry-MAJOR.MINOR.PATCH.<build>.zip` (`.` separator) from `1.0.0` onward; keep the old date-style artifact naming for the one-time `0.9.0` bridge release; default output to `~/.wavefoundry/dist/`
- `check_version.py`: replace `compare_versions()` with `packaging.version.Version`; handle date-string inputs as `"0.0.0"` via explicit regex; raise on unknown formats
- `upgrade_wavefoundry.py`: update version read/compare logic; rewrite hooks that use raw string comparison; auto-discover latest semver zip from `~/.wavefoundry/dist/` when no path given; skip non-matching filenames silently
- `docs/prompts/prompt-surface-manifest.json`: `framework_revision` becomes a semver string
- `.wavefoundry/framework/VERSION`: updated at first semver packaging run (v0.9.0)
- `pyproject.toml`: add `packaging` as a dependency
- Tests and operator-facing docs

**Out of scope:**

- Publishing to PyPI, Homebrew, or any external registry
- Redesigning the zip distribution model
- Calendar versioning or any format other than `MAJOR.MINOR.PATCH`

## Serialization Points

- `_to_version()` must exist before `build_pack.py` and `upgrade_wavefoundry.py` are updated — both depend on it.
- `pyproject.toml` must declare `packaging` before `check_version.py` uses it — land in the same wave as `12tm5-enh python-tool-venv-bootstrap` so the file exists.
- Do not stamp `VERSION` to `0.9.0` until `build_pack.py` is updated — the packager will reject a semver string before that change lands.
- `~/.wavefoundry/dist/` is created by `build_pack.py` on first use; `upgrade_wavefoundry.py` must handle the directory being absent gracefully (fall back to project directory).
- The v0.9.0 bridge artifact naming must be finalized before the upgrade prompt is updated — operator guidance depends on whether legacy adoption uses the old root-zip path or a helper-based migration.

## Risks

| Risk | Mitigation |
| ---- | ---------- |
| Upgrade hooks silently skip for date-based installs if date-string comparison is not handled. | `_to_version()` maps date strings to `"0.0.0"` — all hooks fire for any pre-semver install. Covered by AC-4 and tests. |
| `packaging` library not available before venv is bootstrapped. | Coordinate with `12tm5-enh python-tool-venv-bootstrap` — both changes must land in the same wave. Additionally, `upgrade_wavefoundry.py` may be run with the system Python before the venv exists; use a lazy import with a clear `ModuleNotFoundError` message directing the operator to run `setup_index.py` first. |
| Date-string detection regex too broad or too narrow. | Use explicit `\d{4}-\d{2}-\d{2}[a-z]` pattern in `_to_version()`; raise `ValueError` for unrecognized formats rather than silently mapping to `"0.0.0"`. Covered by AC-10 and tests. |
| Non-matching filenames in `~/.wavefoundry/dist/` cause parse errors in version sort. | `_find_latest_release_zip()` applies the filename pattern before attempting to parse; non-matching files are skipped. Covered by AC-5 and AC-7. |
| Legacy upgrade path fails if the `0.9.0` bridge artifact does not preserve the old date-style zip naming. | Package `0.9.0` with the legacy root-zip filename shape while keeping semver internals; cover this with build-pack tests and explicit operator docs. |
