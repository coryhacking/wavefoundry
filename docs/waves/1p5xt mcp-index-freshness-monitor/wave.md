Title: Mcp Index Freshness Monitor

## Objective

Move continuous index-staleness monitoring **out of the dashboard and into the MCP server**, so the index stays fresh whenever an agent is working regardless of whether the dashboard is running. Today the dashboard watcher is the only continuous monitor (mtime poll + 60s hash-staleness check); git hooks cover commits and MCP mutation tools cover docs saves, but **uncommitted code edits go un-reindexed when the dashboard is off**. This wave adds a dashboard-independent, host-agnostic, in-session monitor in the MCP that reuses the existing single-flight refresh machinery, plus a complementary report/prime on the session-end capture hook. When this wave closes, index freshness no longer depends on the dashboard process.

## Journal Watchpoints

- **Reuse the existing single-flight infra; do not invent a second build path.** `1p5xu` must trigger refreshes through the existing `_start_background_index_refresh` / `_background_refresh_active` lock + `BACKGROUND_INDEX_REFRESH_THROTTLE_SECONDS` throttle so concurrent sessions, the dashboard, and git hooks all coordinate through one lock. A second uncoordinated builder is the failure mode to avoid — treat duplicate concurrent builds as a blocking defect.
- **Per-session scope is intended, not a gap.** The MCP is stdio/per-session, so the monitor runs only while an agent is connected — which is exactly when freshness matters. Do **not** add a separate always-on daemon (that recreates the dashboard-process dependency this wave removes). Git hooks cover commit boundaries between sessions.
- **Fail-safe + cheap.** The monitor thread is a daemon, never blocks MCP startup or queries, swallows its own errors, and uses the stat-fast-path staleness detection (no full hash walk every tick). Config-gated (framework-owned default) so downstream can disable.
- **Dashboard's monitor is removed in this wave (`1p5xw`), but only after `1p5xu` lands.** End state: one continuous monitor (the MCP), the dashboard a read-only health UI. Hard sequencing — never remove the dashboard watcher before the MCP monitor is live and verified (coverage gap). During any transition window the single-flight lock prevents double-building. Accepted tradeoff: "dashboard up, no MCP connected" has no continuous monitor (acceptable — nothing is querying; git hooks cover commits). Preserve the post-upgrade reindex path — verify it survives the dashboard-watcher removal.
- **Synergy with `1p5x8`** — the codebase map's "refresh with the index lifecycle" is driven by this monitor; keep the trigger generic so the map can hook the same refresh signal.
- **`1p5ya` lock consolidation: dedicated `dashboard-server.lock`, flock not existence, delete-after-child-locks.** The lifetime lock is a dedicated `dashboard-server.lock` (rename of `dashboard-process.lock`) — NOT a flock on the metadata JSON (a dedicated never-rewritten `.lock` inode can't be orphaned by a future JSON rewrite). Liveness = flock-try on `dashboard-server.lock` (busy=alive); `start.lock` stays an `flock` (never `O_EXCL`/existence — that would strand a stale lock on crash) and is `unlink`ed only **after** the child holds `dashboard-server.lock`, with the post-acquire double-check retained — otherwise a double-spawn window opens. The unlink is cosmetic/best-effort: a crash-leftover `start.lock` is harmless (unlocked, re-acquired next start). `1p5ya` shares `dashboard_server.py` with `1p5xw` — coordinate same-file edits. Preserve watch-exclusions for both `dashboard-server.json` and `dashboard-server.lock`.
