Title: SQLite Index Substrate

## Objective

Generalize the embedded-SQLite pattern proven by the graph state store into a semantic-index state substrate: one derived-only relational store carrying the decay/freshness sidecar (the storage path `1ro43` recommends), an FTS5/BM25 hybrid lexical layer fused into ranked retrieval, per-path build bookkeeping with `meta.json` preserved as an exported snapshot, and a per-file secret-scan cache that ends the whole-repo re-scan on every ruleset change. When this wave closes, the semantic index has a transactional relational backbone, retrieval gains the documented hybrid-lexical quality lever, and secret scanning gains crash-safe content-addressed incrementality — with zero new dependencies.

## Wave Summary

Three changes on one store. `1rq4h` ships the index-state store (WAL/transactional, schema-versioned, derived-only, drop-and-rebuild recovery) with the freshness/attribution tables that `1ro43` consumes as its decay sidecar, plus a shared maintenance policy (incremental auto_vacuum, end-of-build WAL truncate, `PRAGMA optimize`) and an extension of `wave_index_optimize` into the **unified maintenance verb for every index** — Lance, the index-state store, and the graph store (closing the current gap where the graph index is never reclaimed). `1rrr0` and `1rsha` add resident schemas on that substrate: `1rrr0` — FTS5 lexical tables fused pre-rerank into `search_combined` behind an evaluation gate, plus per-path bookkeeping, the chunk registry with `meta.json` demoted to an exported snapshot, and FTS5 segment-merge maintenance contributed into that unified verb; `1rsha` — a per-file secret-scan cache (content+rules fingerprint skip, transactional, self-healing) shipping Tier 1 with a Tier-2-ready schema so rule-delta scanning is a later additive step. Vectors and chunk content stay in Lance throughout; SQLite holds only derived, rebuildable state, and the secret-scan cache is a skip optimization that never overrides the scanner.

## Journal Watchpoints

- Watchpoint (sequencing): `1rq4h` blocks `1rrr0` — the substrate (schema versioning, recovery, shared helpers) must land before resident schemas are added.
- Watchpoint (single store): exactly one semantic-index state store file; creating any parallel sidecar is a defect. Cross-wave seam with `1ro43` (wave 1ro44): whichever lands second consumes/migrates — coordinate per `1rq4h` Requirement 5 and record the outcome in both change docs.
- Watchpoint (derived-only): storing unrecoverable state in the store is blocked; every table must rebuild from git/Lance/repo, and corruption recovery is drop-and-rebuild with a loud diagnostic.
- Watchpoint (retrieval safety): lexical fusion default-on is blocked until the weak-pattern evaluation is recorded (`1rrr0` AC-3); the kill switch ships regardless; `code_keyword` semantics must not change.
- Watchpoint (reader contracts): `meta.json` readers keep their contract this wave — snapshot byte-compatibility is proven before the store becomes the working source of truth; reader migration is deferred, not smuggled in.
- Watchpoint (graph store stability): shared-helper extraction must leave the graph store's behavior and test suite untouched (`1rq4h` AC-7).
- Watchpoint (schema-version sequencing): `1rrr0` (FTS + bookkeeping) and `1rsha` (secret-scan cache) both add resident tables to the one store; their schema-version bumps must be sequenced, not concurrent, to avoid a version collision.
- Watchpoint (secret-scan safety): the scan cache is a skip optimization only — a differential equivalence harness must prove cache-path findings identical to a full scan, and any corrupt/missing cache fails toward a full re-scan with a loud diagnostic. A cache defect must never become a missed secret. Tier 2 (rule-delta execution) does not ship this wave; only the Tier-2-ready schema and a recorded feasibility spike do.
- Watchpoint (unified maintenance): all SQLite maintenance routes through one path — the shared-helper primitives (`1rq4h` Req 9) and the extended `wave_index_optimize` (`1rq4h` Req 10). FTS segment-merge (`1rrr0` Req 12) and the secret-scan cache (`1rsha`) plug into that path; no change may add a parallel vacuum/checkpoint/optimize surface.
- Watchpoint (unified integrity): corruption verification is one two-layer probe in the shared helper (`1rq4h` Req 11) — structural `quick_check`/`integrity_check` plus source-fingerprint binding for logical staleness — surfaced in `wave_index_health` and routed to the derived-only drop-and-rebuild. FTS `integrity-check` (`1rrr0` Req 13) and the scan-cache fingerprint (`1rsha`) contribute into that probe, not a parallel one. Detection is the whole job; recovery is already free because nothing unrecoverable is stored. Automatic maintenance applies to the index-state store's build path; the graph store gets on-demand coverage only (its build-path behavior stays untouched per `1rq4h` AC-7). The graph store's `-wal` was investigated 2026-07-05 and confirmed NOT growing (build-subprocess-only; the long-lived server never opens the state store, so it checkpoints and drops `-wal`/`-shm` per build) — no graph-store follow-up is needed. The long-lived-reader exposure is real for the new index-state store instead (query-path reads live in the server), which is why Req 9's WAL-truncate is load-bearing there.
