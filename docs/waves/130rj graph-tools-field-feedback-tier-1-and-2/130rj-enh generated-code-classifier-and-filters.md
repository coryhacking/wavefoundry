# Generated-Code Classifier + `exclude_generated` Filter + `generated_node_fraction` Per Community

## Rationale

Aceiss field feedback §6 documented concrete, data-grounded distortions caused by a javacc-generated parser (`ELParser.java`, 3,962 lines) in the graph index:

- **§6.1** Machine-readable generation markers are present in source but not consumed by the indexer.
- **§6.2** All 20 top-betweenness nodes were `ELParser.jj_3R_*` methods. The metric became useless.
- **§6.3** The ELParser Leiden community contained 567 nodes including an unrelated `HibernateInstrumentationModule` (16 nodes) — the community algorithm merged structurally similar but architecturally independent code.
- **§6.4** `jj_scan_token` and `jj_consume_token` appeared in top-20 fan_in alongside genuine project hotspots; agents have no basis to distinguish them.
- **§6.5** Degree max/mean ratio of 35× in generated communities; for handwritten communities this ratio signals an architectural hub. Agents have no signal to know they should not interpret it that way.

Aceiss v2 §6.6 confirmed the right scoping: ship the classifier + tag + `generated_node_fraction` + suppress-if-dominated heuristic. Defer two-layer Leiden as a separate concern; the heuristic captures the common case (ELParser community is 58% generated — well above 0.4 threshold) at near-zero algorithmic cost.

## Requirements

1. **Generated-file classifier** at graph index time. Sources of signal (in priority order):
   - In-file header markers (matched within the first 200 bytes of the file):
     - **Java:** `Generated By:JJTree`, `Generated By:JavaCC`, `Generated from .* by ANTLR`, `Code generated by`, `DO NOT EDIT -- generated code`, `DO NOT EDIT, GENERATED BY`, `@Generated`, `@javax.annotation.Generated`, `@jakarta.annotation.Generated`.
     - **C#:** `<auto-generated>`, `<auto-generated/>` (canonical Roslyn / T4 / source-generator marker; appears inside an XML doc comment like `// <auto-generated/>` or `//------------------------------------------------------------------------------\n// <auto-generated>`), `[GeneratedCode(` attribute (canonical `System.CodeDom.Compiler.GeneratedCodeAttribute`).
   - Path heuristics (directory segments anywhere in path):
     - **Generic:** `generated-sources/`, `build/generated/`, `generated/`.
     - **C#-specific:** `Service References/`, `Connected Services/`.
     - **C#-specific filename suffixes:** `*.designer.cs`, `*.g.cs`, `*.g.i.cs` (Razor / Blazor / WinForms designer files — strong-signal even without header).
   - `.gitattributes` `linguist-generated=true` annotations (parsed once at index time; language-agnostic).
   - Multi-language coverage for Go (`// Code generated by … DO NOT EDIT.`), TypeScript/JS (`// @generated`, `// This file is auto-generated`), Rust (`OUT_DIR/*.rs` build output), Swift (SwiftGen / Core Data generated subclasses), Kotlin (KAPT output), Python (`*_pb2.py` protobuf stubs) is **deferred to a separate follow-up change** (`130rj-enh generated-code-multi-language` or follow-up wave). Operator validation of Java + C# coverage informs whether the multi-language follow-up needs heuristic additions or a different architectural shape.
2. **`generated: true` tag** propagated from classifier to every graph node whose source file matches. Tag is read by downstream filter, fraction, and warning consumers.
3. **`generated_node_fraction` field per community** in the cluster artifact. Computed as `count(nodes where node.generated) / total_node_count`. Surfaced in the `code_graph_community` response and the new `wave_graph_report.communities` section.
4. **Suppress-if-dominated heuristic** in `wave_graph_report.communities`: communities with `generated_node_fraction > 0.4` are flagged `community_type: "generated-dominated"`. They remain in the response but agents can filter on the field. (Aceiss v2 §6.3 — heuristic is a "captures the common case" fix; full two-layer Leiden is deferred.)
5. **`exclude_generated: bool = False` filter** on `wave_graph_report` and `code_graph_community`. When `True`:
   - `wave_graph_report` filters `fan_in`, `fan_out`, `chokepoints`, `betweenness`, and `communities` to exclude generated nodes / generated-dominated communities.
   - `code_graph_community` filters generated nodes from the returned member list.
6. **`betweenness_dominated_by_generated: true` warning** in `wave_graph_report` when >50% of top-N betweenness results have `node.generated == true`. The warning tells agents to skip the metric.
7. **`GRAPH_BUILDER_VERSION` bump** (10 → 11) so existing graph caches re-extract with the new `generated` tag.
8. No false positives on hand-written files containing the word "generated" in a docstring or filename (e.g. `code_generator.py`). The classifier matches against header positions and path segments only — not full-file text.

## Scope

**In scope:**

- `.wavefoundry/framework/scripts/graph_indexer.py`:
  - New helper `_classify_generated(rel_path: str, source_text: str, gitattributes: dict) -> bool`.
  - Header-pattern constants + path-segment constants.
  - `.gitattributes` parser (one-pass at session init).
  - During graph extraction: nodes carry `generated: true` when the classifier matches.
  - Bump `GRAPH_BUILDER_VERSION` 10 → 11.
- `.wavefoundry/framework/scripts/graph_cluster.py`:
  - `read_cluster_payload` surfaces `generated_node_fraction` per community.
  - Community-record builder computes the fraction from node tags.
- `.wavefoundry/framework/scripts/server_impl.py`:
  - `code_graph_community_response`: `exclude_generated` parameter; filter members when True; expose `generated_node_fraction` in the response.
  - `wave_graph_report_response`: `exclude_generated` parameter; filter sections when True; `betweenness_dominated_by_generated` warning when threshold tripped; `community_type: "generated-dominated"` flag on community-overview entries.
  - MCP wrappers updated.
- `.wavefoundry/framework/scripts/tests/test_graph_indexer.py`:
  - Regression tests for each classification path (header, path, .gitattributes).
  - Negative tests for false-positive guard (`code_generator.py`, doc mentioning "generated").
  - `generated: true` propagation to nodes.
- `.wavefoundry/framework/scripts/tests/test_server_tools.py`:
  - `exclude_generated` filter behavior on `wave_graph_report` and `code_graph_community`.
  - `generated_node_fraction` presence on community overview.
  - `betweenness_dominated_by_generated` warning fires when threshold exceeded.

**Out of scope:**

- Two-layer Leiden community detection (Aceiss v2 §6.6 — deferred; heuristic at AC-3/AC-4 covers the common case).
- New `generated:cN` community namespace.
- Operator-tunable threshold for `generated_node_fraction > 0.4` (0.4 is the Aceiss-validated default; expose as a tunable later if other operators report different thresholds work better).

## Decision Log

| Date | Decision | Reason | Alternatives |
|---|---|---|---|
| 2026-05-31 | Heuristic suppression (`generated_node_fraction > 0.4`) instead of two-layer Leiden | Aceiss v2 §6.6 confirmed the heuristic captures the common case at near-zero cost. Two-layer Leiden is a much larger undertaking and can be added later if the heuristic proves insufficient. The ELParser community at 58% generated is well above 0.4 — the threshold catches the Aceiss reported case clearly | Two-layer Leiden detection (deferred to future wave; bigger algorithm change, higher risk, may not improve on the heuristic for the common case) |
| 2026-05-31 | Default `exclude_generated=False` | Backward-compatible. Operators opt in to the filtered view when they want architectural orientation; existing callers see no change | Default True (rejected — changes existing tool behavior; existing snapshot tests and external consumers break) |
| 2026-05-31 | First-200-byte header match (not full-file scan) | Generated-file headers are conventionally at the top of the file. Full-file scan would also match doc-strings and comments mentioning "generated" elsewhere — false positives. 200 bytes covers the canonical header position for all six languages Aceiss surveyed | Full-file scan (rejected — false-positive risk on docstrings). 100 bytes (rejected — too narrow for some grammars with longer copyright headers) |
| 2026-05-31 | `community_type: "generated-dominated"` flag, not silent filter | Operators may still want to inspect generated-dominated communities (e.g. when debugging the indexer itself). Surfacing the flag lets agents decide; `exclude_generated=True` then provides the silent-filter behavior on top | Silent suppression by default (rejected — operators lose visibility into the heuristic's choices) |
| 2026-05-31 | 50% threshold for `betweenness_dominated_by_generated` warning | Aceiss v2 §6.2 confirmed 20 of 20 top-betweenness nodes were generated. 50% is the threshold below which the warning fires conservatively. If half the top-N is generated, the metric isn't useful for architectural analysis | 75% (rejected — too permissive; the metric is already significantly distorted at 50%). 25% (rejected — too aggressive; some legitimate hotspots overlap structurally with generated code) |

## Risks

| Risk | Mitigation |
|---|---|
| Classifier false positives on handwritten files mentioning "generated" | AC-4 explicitly tests for this; header match is positional + bounded to first 200 bytes; path heuristics require directory segments (not substrings) |
| `.gitattributes` parsing adds startup latency | Parse once per session at index init; cache the result. Single-pass over the file — negligible cost |
| Two-layer Leiden deferral could prove insufficient if operator reports more false-positive community contamination | AC-9's heuristic captures the common case (Aceiss validated). If additional reports surface, two-layer Leiden becomes a follow-up wave with concrete evidence to scope it |
| `exclude_generated=True` behavior interacts with the existing `betweenness` skip for large graphs | Both are independent filters applied sequentially; no interaction concerns |
| `GRAPH_BUILDER_VERSION` bump forces full rebuild on upgrade | Standard self-heal pattern; documented as expected post-upgrade behavior |
