# Kotlin Reference Resolution — Enable Tree-Sitter-Backed Identifier Reference Extraction for Kotlin

Change ID: `130tc-enh kotlin-reference-resolution`
Change Status: `implemented`
Owner: Engineering
Status: implemented
Last verified: 2026-06-01
Wave: 130rj graph-tools-field-feedback-tier-1-and-2

## Rationale

Change 130r7 (`130rj-bug java-method-reference-call-sites`) shipped the Java fix for `method_reference` nodes (`Foo::bar`, `this::bar`) classifying as call sites. Implementation inspection confirmed Kotlin's grammar exposes the analogous `callable_reference` shape for `String::length` / `::myFn` / `this::handle`. Adding the entry to `_TS_CALL_PARENT_TYPES["kotlin"]` was dead code because Kotlin wasn't in `_TREE_SITTER_REFERENCE_LANGS` — the reference scanner skipped Kotlin entirely.

130r7 documented Kotlin as deferred:

> "Adding it to `_TS_CALL_PARENT_TYPES["kotlin"]` is dead-code without first adding Kotlin to `_TREE_SITTER_REFERENCE_LANGS` + `_TS_IDENTIFIER_NODE_TYPES` + `_TS_DEFINITION_PARENT_TYPES` (Kotlin tree-sitter reference resolution is not currently enabled at all). Deferred to a separate follow-up that activates Kotlin reference resolution end-to-end."

This change is that follow-up: full Kotlin enablement so `code_callhierarchy` / `code_references` / `code_definition` work for Kotlin codebases the same way they work for Java/C#/JS/TS.

## Requirements

1. **`_TREE_SITTER_REFERENCE_LANGS` add `"kotlin"`** so the reference scanner walks Kotlin source files.
2. **`_TREE_SITTER_DEFINITION_LANGS` add `"kotlin"`** so the definition-resolution pass recognizes Kotlin definitions.
3. **`_TS_IDENTIFIER_NODE_TYPES["kotlin"] = {"identifier"}`** — Kotlin's grammar uses bare `identifier` for symbol references (verified via AST inspection 2026-06-01).
4. **`_TS_CALL_PARENT_TYPES["kotlin"]`** activated with `{"call_expression", "callable_reference"}` (was placed as forward-compatible scaffolding in 130r7).
5. **`_TS_DEFINITION_PARENT_TYPES["kotlin"]`** with `class_declaration`, `object_declaration`, `interface_declaration`, `enum_class_body`, `function_declaration`, `secondary_constructor`, `primary_constructor`. `property_declaration` is **deliberately excluded** — local `val`/`var` shouldn't be classified as a definition for caller/callee navigation, matching the variable-binding-scope pattern handled in wave 130qf for the graph extractor side.
6. **`_TS_IMPORT_PARENT_TYPES["kotlin"] = {"import_header"}`** so `import_header` ancestors classify identifiers as imports.
7. **Tests** cover (a) Kotlin callable-reference call-site attribution (`Helper::process` → non-null line/snippet on incoming entry), (b) traditional Kotlin method invocation (`h.process()`) — sanity check that adding `callable_reference` doesn't break the existing call shape.

## Scope

**In scope:**

- `.wavefoundry/framework/scripts/server_impl.py`:
  - `_TREE_SITTER_REFERENCE_LANGS` + `_TREE_SITTER_DEFINITION_LANGS` add `"kotlin"`.
  - `_TS_IDENTIFIER_NODE_TYPES["kotlin"]` populated.
  - `_TS_CALL_PARENT_TYPES["kotlin"]` activated (entry already drafted in 130r7).
  - `_TS_DEFINITION_PARENT_TYPES["kotlin"]` populated.
  - `_TS_IMPORT_PARENT_TYPES["kotlin"]` populated.
- `.wavefoundry/framework/scripts/tests/test_server_tools.py`: `TestKotlinReferenceResolution` with two tests (callable-reference + traditional-call sanity).

**Out of scope:**

- Kotlin-specific generated-code header patterns (Kotlin doesn't have a canonical generation marker the way Java's `@Generated` or C#'s `<auto-generated>` do; KAPT output lands in `build/generated/` which is already covered by the path heuristic in Change 5).
- Kotlin annotation extraction parallel to Java's `_ts_extract_java_annotations`. Kotlin doesn't have a comparable advice-attribute pattern as the dominant AOP shape; Spring AOP via Kotlin uses Java-style `@Around` which would work IF the file is `.kt` and the grammar surfaces it the same way. Defer to a future operator-reported gap.
- Kotlin coroutine-specific symbols (`suspend fun` resolution, coroutine builders). Out of scope — coroutines are runtime concepts; the graph already captures the static structure.

## Decision Log

| Date | Decision | Reason | Alternatives |
|---|---|---|---|
| 2026-06-01 | Exclude `property_declaration` from Kotlin's definition parents | Matches the variable-binding-scope fix from wave 130qf — local `val r = h.process()` shouldn't push a new scope frame, otherwise the call edge gets attributed to a phantom local-variable scope that short-symbol pruning silently drops | Include `property_declaration` (rejected — reintroduces the wave 130qf bug for Kotlin) |
| 2026-06-01 | Use a single `{"identifier"}` set for Kotlin's identifier node types rather than splitting into property_identifier / type_identifier | Kotlin's tree-sitter grammar uses bare `identifier` for symbol references (verified via AST inspection on `class Worker { val helper = Helper(); fun bar() { helper.process() } }`). No split per-context | Mirror TypeScript's `{identifier, property_identifier, type_identifier}` set (rejected — Kotlin grammar doesn't split that way; the additional types would never match) |
| 2026-06-01 | Land as separate change `130tc-enh kotlin-reference-resolution` rather than amending 130r7 | 130r7's scope was Java-only and explicitly documented Kotlin as deferred pending broader infrastructure. The deferral was the right call given 130r7's narrow bug-fix scope. Treating Kotlin enablement as its own change keeps the audit trail clean | Amend 130r7's ACs (rejected — confuses the audit trail by retconning a deferred decision) |

## Risks

| Risk | Mitigation |
|---|---|
| Kotlin definition-set could miss less-common declaration types (e.g. `companion_object`, `type_alias_declaration`) | Operators on Kotlin codebases will report if specific declarations don't resolve; the change-doc audit trail captures the initial scope explicitly so a follow-up is easy to scope |
| Kotlin file detection (`_detect_language` returning `"kotlin"`) needs to recognize `.kt` and `.kts` extensions | Pre-existing infrastructure; not touched by this change. If `.kts` script files aren't recognized, that's a `_detect_language` gap to address separately |
| The `_treesitter_references` walk for Kotlin uses `respect_ignore=True` by default; vendored Kotlin code under `.gitignore` may be skipped | Standard behavior consistent with other languages; operators who index vendored code use the appropriate workflow-config knobs |
