Title: Expose Index Build Lock Status

## Objective

Give agents an authoritative way to ask "is an index build actually running?" so they stop reading
`.wavefoundry/index/index-build.lock` directly and misreading its by-design persistence as "a build is
in progress." `wave_index_build_status` returns a `lock` object whose `held` is determined by **testing
the real OS lock** — POSIX `fcntl` record lock read via `F_GETLK` (non-destructive, returns the holder
PID); native Windows a sentinel-byte `msvcrt` lock tested via a momentary `LK_NBLCK`. Also fix the
framework's misleading "delete the lock file" recovery message and steer agents off the file via
AGENTS.md + seeds. Plain terminology — no "zombie". This is the new MCP capability that makes the
release a MINOR (1.10.0).

## Journal Watchpoints

- Guard: `framework_edit_allowed` for `indexer.py` + `server_impl.py`; `seed_edit_allowed` for the seed notes.
- Authoritative, not inferential: `held` is determined by **testing the real OS lock** (`F_GETLK` /
  `LK_NBLCK`), never by file presence or a PID heuristic. `classify_index_build_lock_owner` is retained
  only for the reclaim decision inside `_index_build_lock`.
- Concurrency-critical: `_index_build_lock` is the shared acquire path for every builder; the `flock` →
  `fcntl` primitive swap must preserve mutual exclusion + crash-safety — the existing fail-fast
  lock-conflict test plus the full suite gate it.
- Windows: mandatory byte locks — the lock must sit on a **sentinel byte** distinct from the metadata so
  status can still read `owner_pid` during a live build (the `dashboard-server.lock` lesson).
- Terminology: plain language only — no "zombie" (internal `_process_is_zombie` helper name stays).
- Follow-up: this is the MCP capability that justifies the 1.10.0 MINOR bump; ships with the closed
  1p93a + 1p99f fixes.
