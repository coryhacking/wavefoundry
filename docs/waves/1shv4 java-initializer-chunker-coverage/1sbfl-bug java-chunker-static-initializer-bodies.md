# Java Chunker Drops Static/Instance Initializer Bodies (Retrieval Blind Spot for Literal-Rich Init Blocks)

Change ID: `1sbfl-bug java-chunker-static-initializer-bodies`
Change Status: `implemented`
Owner: Engineering
Status: implemented
Last verified: 2026-07-16
Wave: `1shv4 java-initializer-chunker-coverage`
Release target: 1.13.0

## Rationale

Field-confirmed during the 1.12.0 RC retest (2026-07-11, Java consumer repo): the operator's error-string lookup ("Unable to find unambiguous method") missed in BOTH the vector and lexical layers — and the root cause is chunker coverage, not retrieval. The Java chunker never captures static (or instance) initializer bodies: the field repo's `MessagesResourceBundle` chunks contain only the symbols outline, namespace, imports, and the field declaration truncated at `= new Ha…` — the ~40 `messages.put("...", "...")` string literals in the static initializer exist in **no chunk**, so they are invisible to dense retrieval and to the FTS5 lexical layer **by construction**. The 1sbfj backfill repair made the lexical layer complete over what is chunked; this closes the gap in what gets chunked.

The pattern is common in real Java code: message/error catalogs, lookup-map registration, driver/handler registration, and constant tables all live in `static { … }` blocks — exactly the literal-rich content that exact-string lookups (error messages, config keys) target, which is the lexical layer's documented reason for existing.

Re-verified against the current 1.13 candidate on 2026-07-13: the public `chunk_file` path emitted three chunks for a Java class containing static and instance initializer literals, and neither literal appeared in any emitted chunk. The omission exists in both Java implementations: the tree-sitter path selects methods, constructors, nested types, and constants but not initializer nodes; the regex fallback recognizes class declarations and methods but not initializer blocks. This is therefore a chunk-production defect, not a retrieval-ranking defect.

This change should land before 1.13.0. The release already advances the SQLite state-store schema from the 1.12.0 value (`4`) to `6`, whose reset path performs all-layer convergence. Landing the chunk-shape change in that same upgrade avoids a second repository-wide rechunk after 1.13.0. A chunker-only version change is not normally a full re-embed: the indexer rechunks every selected file but reuses embeddings for content-identical chunks, embedding only new or changed chunk text.

## Requirements

1. **Java initializer bodies become first-class chunks across legal type containers:** emit each legal Java `static { … }` and instance `{ … }` initializer block as its own code chunk. The container census must include ordinary and nested classes, enums, and top-level/member records. Records require static-initializer coverage only because Java forbids record instance initializers; that language restriction is an explicit exclusion, not an untested omission. Chunk IDs and breadcrumbs must be deterministic, distinguish static from instance blocks, distinguish multiple blocks of the same kind, and retain enclosing/nested-type qualification. Do not attach initializer bodies to a one-line type declaration or satisfy record coverage through a generic `__module__`/line-window chunk. Initializer chunks are merge-exempt: the existing `_merge_small_chunks` pass must not absorb a short or empty initializer into a declaration, method, or neighboring initializer chunk.
2. **Both Java paths obey the contract without changing Scala fallback behavior:** implement and fixture-pin initializer capture in the tree-sitter path and the regex fallback selected when the Java grammar is unavailable. The supported fallback must not silently restore the retrieval blind spot. Isolate initializer extraction behind a Java-only helper or an explicit `lang == "java"` gate outside the shared Java/Scala parsing behavior; Scala must bypass it unchanged. The Java block-span walk must be a bounded single pass over the source (linear in source length), Java-lexically aware of strings, character literals, comments, escaped quotes, and nested braces, and scope-aware: only direct members of the applicable `class_body`, `enum_body`, or `record_body` qualify. Method/constructor/control-flow blocks, lambdas, anonymous-class bodies, array/object initializers, and nested braces inside an initializer must not be emitted as sibling initializer chunks. Do not use a greedy multiline regex, repeated whole-source rescans, or raw brace counting that can terminate an initializer at data text. Java initializer-scanner failures may use the established generic line-window rescue operationally, but that rescue does not satisfy initializer parity and must be disabled in the parity fixture so the acceptance proof fails closed.
3. **Long field-initializer expressions are reproduce-first and bounded:** reproduce the field-repository symptom with a multiline builder/array/map initializer. If the loss shares the initializer-node extraction mechanism and is localized, retain its literal content within the normal chunk-size limits in this change. If it is a separate, materially broader field-chunking design, record the evidence and explicitly defer it rather than silently expanding this release wave.
4. **Sibling-language census and shared-seam isolation are evidence-driven:** record an executed census for initializer-like constructs. C# static constructors are preliminarily confirmed as captured by the current public path. Kotlin `init` must be tested with its tree-sitter grammar active; a line-window fallback result is not evidence that the grammar-specific path is safe. Because `_chunk_java_like` is currently shared by Java and Scala, a forced-grammar-unavailable Scala golden fixture must pin the complete serialized pre-change fallback chunk surface (`id`, `path`, `kind`, `language`, `lines`, `section`, and exact `text`) and prove that no Java initializer marker/chunk appears. Fix sibling languages only when the same localized mechanism applies; otherwise defer with rationale.
5. **`CHUNKER_VERSION` advances from `31` to `32`:** the shape change must trigger the established rechunk path. Verification must distinguish full rechunk/convergence from vector computation: content-identical chunks reuse embeddings, while new/changed initializer chunks embed.
6. **End-to-end lexical regression:** a Java catalog fixture containing distinctive, tokenizer-stable literals must surface those literals through chunk text, the internal SQLite FTS path, and the registered `code_lexical` tool. `code_search` may be retained as a semantic sanity check, but it is not the proof of exact lexical coverage.
7. **Existing size limits remain authoritative:** oversized initializer blocks must pass through the existing code-chunk split/cap behavior without losing literal-bearing segments or producing one unbounded class chunk.
8. **Raw Java Unicode escapes are outside the initializer-identity lexer boundary:** Java's prelexical `\\uXXXX` translation is a separate mechanism not implemented by this change. A declaration whose keyword or owner depends on that translation may degrade to generic content, but neither path may publish a partial grammar token (for example `u0041`) as a stable initializer owner. Ordinary javac-legal Unicode source characters remain fully supported without normalization.

## Scope

**Problem statement:** literal-rich Java initializer blocks are invisible to all retrieval layers because the chunker never emits their bodies.

**In scope:**

- `chunker.py` Java handling (tree-sitter initializer-node and containing-body selection for classes, enums, and records; reproduction and disposition of field-declaration initializer text, with implementation only if the mechanism is localized under Requirement 3).
- Regex Java fallback parity for initializer blocks, isolated from the shared Scala fallback seam.
- Census of the sibling languages' initializer constructs.
- `CHUNKER_VERSION` bump, chunker fixtures, and FTS/registered-tool integration fixtures.

**Out of scope:**

- Retrieval-layer changes (1sbfj made lexical complete over chunked content; nothing to do there).
- New languages.
- Broad field/member chunking redesign if the long-field symptom proves mechanically separate.
