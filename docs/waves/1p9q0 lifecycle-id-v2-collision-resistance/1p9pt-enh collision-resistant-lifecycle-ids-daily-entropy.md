## Rationale

The current lifecycle-ID encoding is pure time at 5-minute resolution over a 575-year horizon packed into 5 base36 chars (`packed = (days_since_epoch * 288 + bucket_5min) mod 36^5`; `lifecycle_id.py` `_packed_value`/`build_prefix`). Within a single repo/branch it is collision-free — `next_available_prefix` linear-probes past any taken slot, which is why rapid mints come out dense and sequential (`1p9p6 → 1p9p7 → …`). The failure is **cross-branch / distributed minting**: two branches (or agents, or clones) that mint in the same 5-minute window compute the **identical** prefix, because the linear-probe dedup cannot see the other branch's disk until merge. That is a *guaranteed* collision, not a probabilistic one — it is the `1p9hh` collision that motivated wave `1p9jn`.

`1p9jn` added **detection** (ambiguous lookups now surface all candidates). This change is the **collision-rate reduction** counterpart — and the honest framing (per the prepare-council): pure 12-bit hash entropy does **not** *prevent* cross-branch collisions, it turns a *guaranteed* same-window collision into a *rare, still-detectable* one (~1–2 per project/yr at 3–10 minters — the same category as `1p9jn`, at a far lower rate). Only an explicitly-assigned per-minter **node id** guarantees disjointness, and that is deferred (forward-compatible, below). So this change buys a large rate reduction for zero configuration, not a guarantee. It matters more as the framework rolls out across an enterprise with many parallel minters (observed: 3–10 minters per project).

The design insight: the current scheme wildly over-provisions the time axis (575 years at 5-minute resolution) and spends *none* of the 5-char budget on collision resistance. The total information budget of 5 base36 chars is fixed at ~25.85 bits; reallocating the wasted horizon/resolution into low-order **entropy** achieves the rate reduction without widening the ID.

Two provisioning defects compound the waste and must be fixed in the same change:

- **The epoch burns horizon.** Under the new scheme `day_index = days since epoch`, so an epoch in the past starts day_index far from zero and consumes horizon 1:1 with elapsed time. Provisioning today is agent-driven in seeds: `seed-011` step 1.1 (`:46`) computes a fresh-install epoch with `python3 -c "import random; print(random.randint(-365,180))"` — a **nondeterministic, ~5-year-past** epoch (the operator's "random ~5-year start"); `seed-160` (`:166`) backfills the hardcoded `2020-02-02T02:02:00Z` default (`lifecycle_id.py:24`) on upgrade. Under v2 a past epoch is actively wrong (burns ~5 yr; the self-hosted `1999-05-01` burns ~27). The epoch must be set at **provisioning time** and by **code, not agent prose**.
- **No automatic, testable rollout.** Provisioning must be a real **script** — deterministic, idempotent, atomic — that both the seeds and the upgrade pipeline *call*, replacing the inline agent computation. This is the operator's directive and it is the fix for the prepare-council's central finding that `upgrade_wavefoundry.py` has no policy writer and provisioning is agent-driven seed prose (untestable, and running `random.randint`).

## Design summary (all constraints honored)

- **Stay 5 chars**, base36, `value order = time order` for existing **and** new IDs. NOTE (prepare-council F1, code-confirmed): the tree's listing consumers (`list_waves` `server_impl.py:2250`, `list_plans` `:2282`) currently sort by **filename string**, not decoded value. Across the v1→v2 cutover this is safe (both 5-char fixed-width, ASCII alphabet → string order == value order, and OFFSET keeps v2 > v1). But at the 6-char overflow it **inverts** (`"100000" < "zzzzz"` by string). So value-order is **not** free — those sort sites must be rewritten to decode-keyed sorts (AC-6b); the earlier "tools already decode_base36" premise was false.
- **Encoding:** `value = OFFSET + day_index * ENTROPY_SPACE + entropy`, base36, min-width 5, **no modulo**.
  - `ENTROPY_SPACE = 4096` (12 bits) — the clean power-of-2 knee.
  - `day_index = days since a per-repo provisioning epoch` (daily resolution).
  - `entropy = int.from_bytes(hashlib.blake2s((kind + "\x00" + slug).encode("utf-8"), digest_size=D).digest(), "big") % 4096` — **deterministic** and **frozen as a v2 contract** (variant `blake2s`, digest_size, byte order, separator, and UTF-8 encoding all pinned; a golden-vector test — AC-2b — locks the exact mapping so a later swap fails a test instead of silently re-mapping). No RNG, no wall-clock.
  - Horizon ≈ **40 years** (fresh installs, by the offset floor below) to the 6-char overflow.
- **OFFSET** guarantees new IDs sort after all existing ones, stays clear of the reserved `00000` baseline, and sets the visual start band:
  - **Migrated repos:** `OFFSET = scanned_existing_max + MARGIN`, landing in the `1xxxx` band so migrated IDs continue the existing `1p…` sequence. `MARGIN` is a **named constant** sized to cover the intended maximum v1-sibling-branch merge window: the "new > all existing" invariant holds only while a branch still on v1 mints for fewer than `MARGIN / 288` days before merging (v1 climbs ~288/day). `MARGIN` is chosen (and documented) to bound that window (≈1 year of v1 drift, `288 × 366 = 105,408` — operator-tuned from 3 years after the delivery council). The ~2.86M current max is `1p9pk` = 2,858,600.
  - **Fresh repos — scattered `0[1-d]xxx` start with a 40-year horizon floor:** `OFFSET = 36^3 + (blake2s(project_seed) mod (619,520 − 36^3))`, a deterministic per-project scatter into `[36^3, 619,520)` so the first mint reads first char `0`, second char `1`–`d`, then random (never `0000x`), scattered across 572,864 start points, with a guaranteed ≥40.0-year horizon (worst-case day-0 value `619,519 + entropy(≤4095) ≤ 623,616 = 36^5 − 14,610×4096`). `project_seed` captured **once** at provisioning (install instant + project label) and fixed in config. See the start-band analysis below.
- **Graceful 6-char overflow (never wrap):** drop the `% 36^5`. For ≈40 years the value is `< 36^5` (5 chars); when it crosses `36^5` it naturally base36-encodes to 6 chars. A 6-char string can never equal a 5-char one and its value is always `> 36^5 >` any 5-char value → no overlap ever. Order across the boundary is preserved **only** by decode-keyed sorting (AC-6b rewrites the listing sorts); naive string sort inverts. ~1,400 years of 6-char headroom — a safety valve, not a routine mode.
- **Deterministic-node forward compatibility:** the 12-bit field supports a later config-only carve of the top 4 bits as an explicitly-assigned `node_id` (0–15) leaving 8 bits of hash. Concrete layout: `entropy = (node_id << 8) | (blake2s(...) mod 256)` **only when node assignment is enabled**; **node unset = full 12-bit hash** (today's behavior) — a hard 4/8 split from day one would silently narrow entropy 4096→256 and quadruple the collision rate. Carving later does not change already-minted values (immutable strings; entropy stays in `[0,4096)` → day-bands unchanged) but is a contract change to the entropy mapping, so it must be `node_bits`-flag-stamped in policy, not silent.
- **Linear-probe kept** as the same-repo tiebreaker on top of the base value (dense, ordered same-branch mints unchanged). This means a v2 ID is deterministic *absent a local same-day same-bucket collision* — on a ~1%/day collision (12-bit space at ~10 mints) the probe advances and the stored ID = base+1, so cross-machine reproducibility is "same base value absent a probe collision" (scoped in AC-2).

### Start-band analysis (why a 40-year floor)

First char `0` fixes value `< 36^4`; the second char picks which `36^3`-wide sub-band the OFFSET scatters in. Value grows `4096/day` ≈ 1.5M/yr, so a fresh project climbs one band every ~11 days and rolls to `1xxxx` after ~1 yr regardless — the offset only sets *where it starts*. The band top trades scatter against the horizon-to-overflow floor; the chosen cap is the offset at which the worst case is exactly 40 years, `36^5 − 14,610×4096 = 623,616` (offset top `619,520` after entropy headroom):

| Fresh 2nd char | OFFSET band | Scatter (start points) | Min horizon to 6-char |
| --- | --- | --- | --- |
| `1` only | `[46,656, 93,312)` | 46,656 | 40.4 yr |
| `1–5` | `[46,656, 279,936)` | 233,280 | 40.2 yr |
| `1–9` | `[46,656, 466,560)` | 419,904 | 40.1 yr |
| **`1–d` (40-yr floor)** | **`[46,656, 619,520)`** | **572,864** | **40.0 yr (exact)** |
| `1–z` (full `0`-band) | `[46,656, 1,679,616)` | 1,632,960 | 39.3 yr |

The horizon "burn" across the whole `0`-band is only ~1 yr of ~40, and the boundary is the *graceful* overflow, so it is low-stakes; the `1–d` cap is the offset at which the worst case is exactly 40.0 years — a principled floor guaranteeing every fresh project ≥40 yr with 572,864 start points (ample scatter). Migrated repos are unaffected (their `scanned_max + MARGIN` offset lands in `1xxxx`). Mirrors the `131bt` epoch-shift precedent.

## Requirements

1. **New `v2` encoding.** `value = OFFSET + day_index * 4096 + entropy`, base36, min-width 5, **no `% 36^5`**. `day_index` = whole days from the configured epoch; `entropy` = the frozen `blake2s` expression above. Decoded order = time order among new IDs; every new ID exceeds every existing ID (via OFFSET).
2. **Frozen entropy contract.** The exact mapping — `blake2s` variant, `digest_size`, `int.from_bytes(..., "big")` byte order, `kind + "\x00" + slug` UTF-8 encoding, `% 4096` extraction — is part of the `v2` contract and is locked by a golden-vector test (a fixed `(kind, slug, day, epoch, offset)` → a hardcoded expected prefix *string literal*). A later hash/variant/byte-order/separator change must fail that test, not silently re-map new IDs while `scheme_version` stays `v2`.
3. **Provisioning by a code script (operator directive).** A deterministic, idempotent, atomic **script/helper** — not agent prose — computes and writes the policy. Compute logic (`scan_max → OFFSET`, epoch, `project_seed → offset scatter`) lives in `lifecycle_id.py` (unit-testable, reusing `_existing_prefixes`); file-write orchestration lives in a new `materialize_lifecycle_policy(root)` phase in `upgrade_wavefoundry.py`, mirroring the `materialize_secrets_policy` Phase-2b precedent (`:2045`/`:2464`) — **phase placement and status-line shape only** (prepare-lane pin: the precedent is create-only + non-atomic `write_text`, so its mechanics must NOT be copied). Concretely: (i) **read-modify-write** — load the full `docs/workflow-config.json`, replace only the `lifecycle_id_policy` value, preserve every other top-level key and re-serialize the whole document; (ii) write via **same-directory temp + `os.replace`** (the `indexer._atomic_write_text` pattern incl. the Windows sharing-violation retry — never a `/tmp` temp file, which risks cross-device rename); (iii) if the existing file is **unparseable JSON, fail loudly and write nothing** — never replace a corrupt-but-recoverable operator file. Idempotence is keyed on `scheme_version == "v2"` presence (all-or-nothing — a partial prior write is re-attempted). The script gets a **first-class CLI entry** (a `--materialize-lifecycle-policy` flag on `upgrade_wavefoundry.py` that runs only this phase and exits) so seeds invoke a named framework command, not inline `python3 -c` prose. The install seed (`seed-011`) and upgrade seed (`seed-160`) are rewritten to **call exactly that command** (retiring `seed-011`'s `random.randint` past-epoch and `seed-160`'s `2020-02-02` backfill, and fixing seed-160's stale cross-reference to the retired seed-011 Step 1.1 formula); these seed edits are behind the `seed_edit_allowed` gate and must stay self-contained (no internal wave/change/ADR IDs).
4. **Provisioning-time epoch.** Install → `epoch = install date`; upgrade → `epoch = rollout date` (day_index ≈ 0, full ~40-yr horizon). Retire `DEFAULT_EPOCH_UTC = 2020-02-02` as a primary path; the residual code fallback is compute-or-error, never a stale fixed year. Resolve the `seed-160` "never overwrite `epoch_utc`" invariant vs the migration writing a new epoch: v2 reuses `epoch_utc` and the migration is gated by `scheme_version` absence (a repo already `v2` is never re-epoched).
5. **OFFSET selection.** Migrated: `OFFSET = scanned_existing_max + MARGIN` (`MARGIN` a named constant sized for the max merge window, documented). Fresh: `OFFSET = 36^3 + blake2s(project_seed) mod (619,520 − 36^3)`. Both clear the reserved `00000`, guarantee new > all existing (bounded by `MARGIN` — see Risks), and are written by the provisioning script.
6. **Graceful 6-char overflow + decode-keyed ordering.** Never wrap; widen to 6 chars naturally past `36^5`. Ordering-sensitive consumers must compare by **decoded integer value** — concretely, rewrite `list_waves` (`server_impl.py:2250`) and `list_plans` (`:2282`) from `sorted(...)` on names to `sorted(..., key=<decoded prefix>)`. Emit a loud operator warning as the horizon nears (< ~2 yr of 5-char space) for a deliberate re-issue.
7. **ID-format contract audit (width AND ordering AND validators).** Enumerate and reconcile **every** ID-ordering + ID-format-contract site, not just width literals. Named sites (prepare-council, code-confirmed): `_PREFIX_RE`/`rjust(5)` (`lifecycle_id.py`) → min-width; `LIFECYCLE_PREFIX_PATTERN` (`wave_lint_lib/constants.py:216`, feeds 5 docs-lint validators) → `{5,6}`; `_SEC_ID_RE` (`secrets_validators.py:471`); the `prefix_width must be 5` validator (`core_validators.py:45-47`, a hard `!= 5` conflict with 6-char) → reconcile; the width-4 build sites (`build_pack.py` `[-4:]` and the `[A-Za-z0-9]{4}` collision-warning regex at `:144` — which goes vacuously silent if the suffix re-bases); the two decode-keyed sort rewrites (Req 6). Extend `_check_lifecycle_id_policy` (`core_validators.py:18`) to validate the new v2 keys. Prepare-lane additions to the audit: three more seeds ship **normative pre-131bt ID-format prose** ("4 Crockford Base32 digits for hours since epoch + one minute-bucket character" — already wrong for v1, doubly wrong under v2) at `seed-170:134`, `seed-001:33`, `seed-110:60` — correct them under the same `seed_edit_allowed` gate; and two sites recorded as **justified, not rewritten**: the `wave_dir.rglob` match-lookup sorts (`server_impl.py:4100/4263`, determinism-of-match only) and `wave_validators.py:875` `sorted(plans_root.iterdir())` (deterministic output order, not a time-order contract).
8. **build_pack suffix — own pure-time function, fully decoupled from the v2 mint.** The build number is a temporally-sortable token for release artifacts; it needs none of v2's collision-resistance, entropy, offset, or daily bucketing. `compute_build_suffix`/`_current_build_suffix` must **stop routing through `build_prefix`** (which under v2 carries offset+entropy and daily resolution) and instead compute a **standalone pure-time index**: `base36((days_since_build_epoch × 288 + bucket_5min) mod 36^4)` — the current v1 packing (288/day, **5-minute resolution** so consecutive same-day builds still differ, `36^4/288 ≈ 5,832 days ≈ 16-yr` horizon per lap), on a **stable build epoch**. Prepare-lane pin: the build-epoch constant is **`1999-05-01T00:00:00Z`** — the current effective epoch (`_current_build_suffix` reads the self-hosted repo policy today), so suffixes stay value-identical with every shipped pack (`…1.10.0.p9cr.zip` etc.); a fresh epoch would re-base new suffixes below shipped ones. The inherited lap boundary (`mod 36^4` wraps ≈ 2031-04 from this epoch) is the status quo, documented in the code comment — cross-version ordering rides semver, and same-semver repackages ride the mtime tie-break, so the lap boundary is cosmetic. The suffix is **min-width-4** (`rjust(4)` — a post-wrap value below `36^3` must not emit 3 chars). Rejected alternatives (see Decision Log): last-4 of the v2 value (wraps every ~410 days under 4096/day + entropy-polluted); first-4 of the v2 value (monotonic ~40 yr but v2's *daily* resolution collapses same-day builds to one suffix — a regression from v1's 5-min buckets — and the offset dominates the visible chars). Also fix the width-pinned `[A-Za-z0-9]{4}` collision-warning regex (`build_pack.py:144`) — it works today but goes silently vacuous if the suffix width ever changes; make it width-tolerant so it stays live.
9. **Automatic, idempotent, atomic migration** (delivered by the Req-3 script): scan max → compute OFFSET+epoch → atomic write of a `scheme_version: v2` `lifecycle_id_policy` block; migrate `v1 → v2` once, keyed on `scheme_version` presence; re-run is a no-op; existing IDs never rewritten; crash mid-write leaves valid JSON (temp+rename) and re-attempts.
10. **Dual-scheme code, atomic per-repo cutover.** `lifecycle_id.py` reads `scheme_version`, `epoch`, and `offset` as one policy tuple (read once per mint): `v2` → new encoding; absent/`v1` → current encoding unchanged. New code + un-migrated config keeps minting `v1` until the script writes `v2`. Widen the loader tuple `(epoch, hour_offset)` → `(epoch, hour_offset, offset, scheme_version)` — **retain `hour_offset`** so the v1 encoder (which applies `hour_offset * 12` buckets, `lifecycle_id.py:171-172`) stays byte-unchanged (v1 ignores `offset`/`scheme_version`; v2 ignores `hour_offset`). Prepare-lane pins: (i) **`policy=` accepts both the legacy 2-tuple and the widened 4-tuple** (length dispatch) — ~20 existing test call sites pass 2-tuples, and `build_pack`'s `policy=(epoch_utc, 0)` caller is only one of them; (ii) **failure modes**: when the `scheme_version` key is *present*, validate strictly and raise `ValueError` (the existing `hour_offset` pattern) on `scheme_version` outside `{v1, v2}`, or under `v2` on `offset` missing/bool/non-int/negative-or-below-`36^3`, or `epoch_utc` missing/invalid — never silently default `offset` to 0 or the epoch to `DEFAULT_EPOCH_UTC` (a defaulted offset mints in/below the reserved band; a silent v1 revert on a migrated repo mis-orders every new ID). Silent v1 fallback remains only for genuine absence of the policy block; a file-level `JSONDecodeError` additionally emits a warning naming the parse failure; (iii) **compute-or-error is scoped to the v2/provisioning path** — the v1 no-config fallback (`DEFAULT_EPOCH_UTC`) stays byte-unchanged (live callers mint without policy on un-provisioned repos), and the `DEFAULT_EPOCH_UTC` symbol is retained (demoted, not removed — six test sites reference it).
11. **Config key names + schema.** Reuse `epoch_utc`; add `offset`, `scheme_version` (and `node_bits` reserved). Pin exact key names; keep the block backward-compatible so v1 repos and existing tests are unaffected. `project_seed` (fresh installs) is stored under exactly that key name (no `key`/`token`/`secret` substring) as the **raw human-readable string** (install instant + project label), not a derived hex/base64 digest — pinned so the secrets scanner's generic-api-key keyword set and entropy heuristics never fire on it. Extend `_check_lifecycle_id_policy` (`core_validators.py:18`) with the same strict v2 rules as the loader (Req 10 ii) so docs-lint catches hand-edits before a mint does.
12. **Preserve success behavior.** Same-repo single-branch minting stays dense/ordered via the retained linear-probe; the CLI/MCP mint surfaces keep their contracts; the v1 path is byte-unchanged.

## Scope

**Problem statement:** The 5-char lifecycle-ID scheme spends its entire budget on a 575-year/5-minute time axis and none on collision resistance (guaranteed same-window cross-branch collision). Provisioning is agent-driven seed prose that sets a nondeterministic past epoch (burning horizon), there is no code migration, and multiple ID-ordering/format-contract sites assume fixed 5 chars.

**In scope:**

- `v2` encoding + frozen `blake2s` entropy + dual-scheme dispatch in `lifecycle_id.py`; no-wrap 6-char overflow; retained linear-probe.
- A provisioning **code script** (compute in `lifecycle_id.py`, orchestrate `materialize_lifecycle_policy` in `upgrade_wavefoundry.py` per the secrets Phase-2b precedent) — deterministic, idempotent (`scheme_version`-keyed), atomic (temp+rename) — computing/writing epoch + OFFSET (migrated and fresh) + `scheme_version`.
- Rewrite `seed-011` (install → fresh repos auto-provision via `wf setup` Step 0; retire `random.randint`/past epoch) and `seed-160` (upgrade → audit-only; retire `2020-02-02` backfill), **behind `seed_edit_allowed`**; `setup_wavefoundry.py` Step 0 (`_provision_lifecycle_policy_if_absent`, repo-root anchor guard) + `install-log.template.md` rows 1.1/1.2 + the `phase_cleanup` reconciliation backstop (`_ensure_lifecycle_policy_backstop`) — both operator-directed during delivery.
- The full ID-format/ordering audit (Req 7): decode-keyed sort rewrites (`server_impl.py:2250,2282`), `LIFECYCLE_PREFIX_PATTERN`/`_SEC_ID_RE`/`prefix_width==5`/`_check_lifecycle_id_policy` reconciliation, min-width `_PREFIX_RE`/`rjust`, width-4 build sites.
- `build_pack` suffix re-base (decoupled, pure-time, horizon-preserving, 6-char-safe) + the vacuous `{4}`-warning fix.
- Config-schema keys + loader tuple widening + `build_pack` policy-caller audit.
- The forward-compatible node-carve layout documented + `node_bits`-flag-stamped (built later, not now).
- Tests (see ACs): golden vector, encoding/order/overflow/offset/epoch/margin/determinism/migration-idempotence-atomicity/dual-scheme/sort-order/width-tolerance/suffix.

**Out of scope:**

- Building explicit node-id assignment now (layout + flag reserved; assignment surface is a separate change).
- Widening the primary width to 6 as the normal case (5 is hard; 6 is the overflow valve only).
- Re-minting/rewriting existing IDs; changing the `00000` legacy semantics.
- `1p9jn` detection (remains the backstop for the residual rare collision).
- Cross-repo absolute-date comparability (never a guarantee; per-repo provisioning epoch).

- [x] AC-2: Deterministic base encoder — the same `(kind, slug, day, epoch, offset)` yields the same **base** prefix across processes, tested at the `build_prefix`/entropy-helper level (NOT through `next_available_prefix`, which linear-probes); reproducibility is scoped "absent a local same-day probe collision"; distinct-tail is asserted with pre-verified golden slugs (not universal injectivity — 4096 buckets birthday-collide). | Migration mints an ID that sorts before an existing one because a long-lived v1 sibling branch out-climbs the offset. | The "new > all existing" invariant is **time-bounded by `MARGIN`**, not absolute: a v1 branch minting for `> MARGIN/288` days then merging exceeds the early v2 band. `MARGIN` is a named constant sized (and documented) to cover the intended max merge window (≈1 yr of v1 drift, operator-tuned); AC-3 asserts `offset ≥ scanned_max + MARGIN`. Beyond that window is an accepted, documented bound. |
| Silent re-map of the entropy mapping under a stable `scheme_version`. | AC-2b golden vector freezes `blake2s`/digest_size/byte-order/separator/encoding; any drift fails a test. |
| Partial/crash-mid-write migration leaves invalid or half-written config. | Atomic temp-file+rename; idempotence keyed on `scheme_version` presence (all-or-nothing), so a partial prior write is re-attempted, not skipped; AC-7 tests the partial-write path. |
| A missed ID-format/ordering site breaks lookups/validators at the 6-char overflow (or the `prefix_width==5` validator fails immediately). | AC-6a/6b require an audit oracle covering SORT + VALIDATOR + WIDTH sites (named), each widened/reconciled; a synthetic 6-char ID + mixed corpus exercised now. |
| Build-suffix horizon collapse from a naive re-base. | AC-9 requires a decoupled pure-time horizon-preserving suffix with a multi-year monotonicity test; the vacuous `{4}` warning fixed. |
| Node-carve later narrows today's entropy. | "node unset = full 12-bit hash" pinned; the 4/8 split is a masked overlay applied only when `node_bits` is set; carving is `node_bits`-flag-stamped (not silent) and does not change already-minted immutable values. |
| Deterministic entropy makes same-`(kind,slug)`-same-day a hard duplicate. | Desirable (true duplicate); the linear-probe advances a genuine same-repo re-mint; cross-repo caught by `1p9jn`. AC-2 reproducibility is scoped "absent a local probe collision." |
