Title: Lifecycle Id V2 Collision Resistance

## Objective

Replace the lifecycle-ID encoding with a collision-resistant `v2` scheme so independent same-window mints on different branches no longer produce identical prefixes (the guaranteed cross-branch collision that `1p9jn` could only detect). When this wave closes, new IDs carry a daily time index plus 12-bit deterministic entropy in a fixed 5 chars (lex/value order preserved for existing and new), fresh installs get a scattered `0[1-d]xxx` start with a guaranteed ≥40-year horizon, the scheme never wraps (graceful 6-char overflow), the epoch is set at provisioning time so no horizon is burned, and the rollout migrates automatically and idempotently on upgrade.

## Journal Watchpoints

- **High blast radius:** this rewrites a core primitive and touches every `[0-9a-z]{5}` / `rjust(5)` assumption across regexes, validators, dashboards, filename parsers, and `build_pack`. Watchpoint: the 5-or-6-char tolerance audit (AC-6) is the highest-risk workstream — a missed fixed-5 site stays latent until the 40-year overflow triggers, so the audit list is the oracle and must be exhaustive.
- **Migration ordering guard:** OFFSET must clear each repo's scanned max; the migration must be idempotent (v1→v2 once, no re-shift on re-run) and must never rewrite existing IDs. Verify against a fixture repo before trusting the upgrade path.
- **Dual-scheme atomicity:** code keys off `scheme_version`; a partially-upgraded repo (new code, old config) must keep minting v1 until migration writes v2 — verify no surprise mid-flight shift.
- **Consider phasing at implementation:** landing the v2 encoder + dual-scheme first and the 6-char tolerance audit as a tightly-reviewed second pass may de-risk the audit; decide at pre-implementation.
- **Determinism:** entropy and the fresh-install offset seed must be deterministic (blake2, no RNG/wall-clock in the hot path); the only clock read is the once-at-provisioning epoch/seed capture.
