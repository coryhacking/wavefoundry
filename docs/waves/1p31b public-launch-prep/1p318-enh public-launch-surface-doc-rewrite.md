## Rationale

Wavefoundry is being prepared for public push to GitHub. The current visitor-facing surface documentation was written for an internal-development audience and carries multiple credibility problems that will damage first-contact impressions:

- **Stale version badges.** `README.md` says `1.0.1` in three places; current `VERSION` is `1.3.31+p30q`. `docs/references/project-overview.md` references VERSION `2026-05-06g`, also stale. For a *framework about structured delivery*, version-prose drift is the worst possible first impression — it directly contradicts the project's selling point.
- **Mechanism-first framing.** Opening sentence loads four unfamiliar abstractions ("agent harness," "feedforward guidance," "numbered seed prompts," "feedback sensors") before any concrete value lands. A Hacker News drive-by reader closes the tab.
- **Missing "your first wave" walkthrough.** The operator explicitly called this out. README explains *what* a wave is conceptually but never walks a new user through `install → docs review → first wave → close`.
- **No transcript or example.** The catalog has 47 MCP tools and ~25 public command phrases. Reader cannot tell whether "Plan feature" is one phrase that works or a complex protocol they need to learn.
- **Audience uncertainty.** Multiple audiences land on the README — solo dev from HN, engineering manager evaluating tools, security-conscious enterprise developer, OSS maintainer testing "is this just a wrapper around X" — and the current structure addresses none of them directly.
- **Self-hosting evidence hidden.** The single strongest credibility statement ("Wavefoundry uses the Wave Framework to develop itself") gets one sentence and is never *shown*. A link to a real closed wave does more credibility work than any prose.

This change rewrites the public-launch surface (README + the visitor-facing parts of `docs/prompts/index.md` and `docs/references/project-overview.md`) and adds three missing artefacts (file-write disclosure, host first-class-vs-compatible split, self-hosting proof link). Out of scope: `AGENTS.md` (agent-facing, not visitor-facing) and `CONTRIBUTING.md` (already in good shape; spot-check only).

## Approach

Adopt the maintainer's proposed structure (hero → pitch → value → quick start → "your first wave" walkthrough → installed footprint → concepts → host table → principles → non-goals → status/license) as v1, informed by the red-team's three structural alternatives (OC-001 transcript-first, OC-002 concept-spine diagram, OC-003 two-doors audience-routing) as targeted additions rather than wholesale replacements:

- **From OC-001 (transcript-first):** the "your first wave" walkthrough is a **literal three-block transcript**, not abstract prose. The Close-wave step intentionally fails on first attempt because a required lane is missing — this single design choice rebuts the "just a wrapper around prompts" suspicion structurally rather than rhetorically.
- **From OC-002 (concept-spine):** ship **one canonical diagram** showing change-doc → wave → prepare gate → implement → review lanes → close gate, with sensor and seed callouts. Mermaid (renders on GitHub) is the format target.
- **From OC-003 (two-doors):** keep linear structure but add a **2-line entry-router** above the quick start: "If you're here to try it, jump to Quick Start. If you're here to evaluate it for your team, jump to For teams." Cheap addition; serves both primary audiences explicitly without forcing a forking page structure.

The red-team's top-3 leverage actions (transcript-with-failed-gate, symptom-led opening, three honesty blocks) are baked into the requirements below.

## Requirements

1. The README opens with the **symptom of the problem**, not the mechanism. Concrete draft (red-team IR-001): *"Your AI coding agent forgets what it did last session, skips review steps you told it to follow, and ships work that drifts from the plan. Wavefoundry is a local harness that makes the agent plan, review, and close work in tracked units — using Markdown files in your repo and an MCP server that runs on your machine. No service, no account, no telemetry."*
2. A `Who this is for / who this is not for` block follows the opening pitch and explicitly de-selects wrong-audience visitors so they bounce cleanly rather than carrying the wrong expectation through the rest of the page.
3. The `Your first wave` walkthrough is a **three-block transcript** with annotated callouts. Block 1: agent receives `Plan feature: <example>` and produces a change doc (excerpt). Block 2: `Create wave` → `Add change to wave` → `Prepare wave` shows readiness output. Block 3: `Implement wave` → `Review wave` → `Close wave` → **the first close attempt is refused by a structural gate** (e.g., missing operator signoff or unrecorded lane); the second close succeeds. Caption on the refusal: *"This refusal is enforced in Python, not in the seed prompt. The agent cannot talk its way past it."*
4. A `What Wavefoundry writes to your machine` block lists every file path Wavefoundry creates or modifies (in-repo + `~/.wavefoundry/`) with one-line justification per path. Network footprint stated as: **"zero at runtime to Wavefoundry-controlled endpoints. Embedding model weights are fetched from Hugging Face on first index build (cached locally thereafter). No telemetry, no account, no service."** (Per F-SR-1; the earlier "zero at runtime" phrasing was not literally true for first-run weight fetch.)
5. A `What it looks like installed` block shows the file-tree **delta** the install adds to a target repo, marked as committed vs. gitignored vs. user-owned. **Pre-publish verification required:** the claim "Wavefoundry never edits files outside `.wavefoundry/`, `docs/`, `.mcp.json`, `.claude/`, `.cursor/`, `.codex/`, `.junie/`, `AGENTS.md`, `CLAUDE.md`" must be confirmed in two distinct passes: `architecture-reviewer` verifies the claim against observed behavior (and against `docs/architecture/data-and-control-flow.md` as the authoritative source); `security-reviewer` verifies the claim's framing for trust-boundary precision. The verifier scope explicitly includes `~/.wavefoundry/` cache directory contents (no credentials, no PII).
6. The agent-host list **explicitly splits first-class hosts** (Claude Code, Cursor, Codex, Junie — one-command install via `render_platform_surfaces`) from **compatible hosts** (Copilot, Windsurf, Air, Warp — standard MCP stdio entry, no host-specific rendering). The split is stated honestly in the host-support table.
7. A `Built with Wavefoundry` proof block links a real closed wave with its actual readiness review, council signoffs, and shipped artefacts. Recommended candidate: `docs/waves/1p2q3 field-feedback-round-4/wave.md` (this wave's parent), which exercises the full lifecycle including red-team review.
8. The `47 MCP tools / four surfaces` headline is replaced with: *"You'll use about six phrases day-to-day; the rest are there when you need them."* The six phrases (named explicitly per F-DC-1): **`Plan feature`**, **`Create wave`**, **`Add change to wave`**, **`Prepare wave`**, **`Implement wave`**, **`Close wave`**. List them in the walkthrough; mention 47 only in a `Full tool surface` footnote with a link to `docs/prompts/index.md`.
9. The version badge syncs automatically from `.wavefoundry/framework/VERSION` (either via a build-pack post-stamp hook or a release-derived shields.io badge). Manual prose-version sync is removed entirely so this defect cannot recur.
10. The `docs/prompts/index.md` framing copy at the top of the file is rewritten to be visitor-friendly (drop seed-IDs and internal jargon from the opening paragraph; preserve the command table verbatim **and preserve the Legacy Aliases table verbatim** — that table is a behavioral contract agents route off; per F-DC-2).
11. `docs/references/project-overview.md` is refreshed: stale VERSION reference replaced, decide whether this doc stays internal or becomes a redirect to README for visitors.
12. `AGENTS.md` and `CONTRIBUTING.md` spot-checked for version-staleness; no structural rewrite required.
13. One canonical Mermaid diagram (concept spine) sits between the walkthrough and the deeper Core Concepts section. Source format is Mermaid so GitHub renders it natively. Diagram source is checked in alongside the README.
14. An entry-router (2 lines) sits above the Quick Start: "If you're here to try Wavefoundry on a project → Quick Start. If you're here to evaluate Wavefoundry for your team → For teams."
15. A `For teams evaluating Wavefoundry` short section answers the four manager-evaluation questions explicitly: adoption cost, lock-in story, security story, current-usage status. The current-usage answer is honest: single maintainer, used in production for the framework itself, no external case studies yet.
16. **An explicit install walkthrough lives in Quick Start**, separate from the "Your first wave" lifecycle transcript (Req-3). Format: literal commands + named operator-visible signals covering all six steps from Python-version check to verified MCP. Steps walked, in order: (a) verify Python 3.11+ is available — expected signal: `Python 3.11.x` (or higher) from `python3 --version`; (b) download the **latest stable release zip** from GitHub Releases and drop it at the target-repo root — disqualifying patterns named explicitly: pre-release tags (`-rc`, `-beta`, `-alpha`), branch-zip downloads from `Code → Download ZIP`, and any asset not attached to a published Release; (c) tell the agent `Install Wavefoundry` — expected signal: agent confirms zip unpacked with `.wavefoundry/`, `docs/`, and `.mcp.json` present at repo root; (d) run `python3 .wavefoundry/framework/scripts/setup_wavefoundry.py` — expected signal: final output reports the index-ready summary line (chunks indexed, layer counts); (e) register the MCP server with the host — **walkthrough completes inline for Claude Code via `render_platform_surfaces --platform claude`**; **other hosts hand off to `docs/prompts/install-wavefoundry.prompt.md` at a labeled hand-off point in the walkthrough with a labeled resume signal (host shows the `wavefoundry` server in its MCP list)**; (f) verify via `wave_index_health()` (expected signal: `semantic_ready: true`) and `wave_audit()` (expected signal: `ready: true`). The lifecycle transcript (Req-3) picks up after step (f) completes.
17. **The prerequisite block sits above the first install command** and lists all three hard prerequisites: (i) **Python 3.11+** with a copy-pasteable `python3 --version` check; (ii) **macOS, Linux, or Windows-via-WSL2** host OS; (iii) **an MCP-aware agent host** (Claude Code, Cursor, Codex, Junie, Copilot, Windsurf, Air, or Warp). The Python-version branch reads: *"Do not proceed past this block until `python3 --version` reports 3.11 or higher; install via your package manager or `pyenv`, then resume at the install walkthrough's step (b)."* The imperative is testable and the resume point is named so the reader is not abandoned. No install instruction appears in the document before all three prerequisites are visible. The Python check is repeated as the first line of the install-walkthrough preamble (Req-16) so a reader who jumps into the walkthrough from the entry-router still encounters it.

## Scope

**Problem statement:** The current visitor-facing surface documentation fails on first contact with multiple audiences. Stale versions, mechanism-first framing, missing walkthrough, missing evidence, and uneven host claims combine to set the wrong impression of a project that's actually in production and structurally sound. Public push without this rewrite damages launch reception.

**In scope:**

- `README.md` — full rewrite per the structure above.
- `docs/prompts/index.md` — framing copy at the top only; command table preserved.
- `docs/references/project-overview.md` — version refresh + internal/visitor disposition decision.
- One new canonical Mermaid diagram (committed alongside README).
- One Pre-publish architecture/security verification pass on the "files Wavefoundry writes" claim (Req-5).
- Light spot-check of `AGENTS.md` and `CONTRIBUTING.md` for version-staleness.

**Out of scope:**

- Restructuring `AGENTS.md` (agent-facing entry; serves its current audience well).
- Adding screenshot assets (could be a follow-on if A/B testing shows transcript alone underperforms).
- Building external case studies (red-team flagged honesty about "no external case studies yet" as a feature, not a defect, for v1).
- Build-pack automation for version-badge sync — `Req-9` calls for it but the implementation can land in a sibling change if it's bigger than expected.
