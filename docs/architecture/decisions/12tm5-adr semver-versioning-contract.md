# 12tm5-adr — Semver Versioning Contract

Owner: Engineering
Status: accepted
Last verified: 2026-05-22

## Context

Wavefoundry uses `YYYY-MM-DDx` date strings (e.g., `2026-05-20i`) as its release identifier in `.wavefoundry/framework/VERSION`, `framework_revision` in `docs/prompts/prompt-surface-manifest.json`, zip artifact filenames, and upgrade ordering. This format has two structural problems:

1. **Weak public contract.** Date strings do not communicate compatibility intent, expected upgrade scope, or the difference between patch, minor, and major changes — making them unsuitable for a public operator-facing release identifier.

2. **Broken comparison.** `compare_versions()` in `check_version.py` uses lexicographic string ordering, which works for date strings but silently breaks for semver (`"1.10.0" < "1.9.0"` lexicographically). Upgrade hooks in `upgrade_wavefoundry.py` gate migration steps using raw string comparison (`ctx.from_version < "2026-05-19a"`), which will fire incorrectly once semver strings appear.

Adoption of semver was decided in wave `12t9b` before the first public release. The implementation is tracked in `12tm5-enh migrate-versioning-to-semver`.

## Decision

Wavefoundry adopts `MAJOR.MINOR.PATCH` semver for all externally visible release identifiers. The migration is two-phase:

- **v0.9.0 (bridge release):** First semver-format pack internally, but packaged with the old date-style zip artifact name (`wavefoundry-YYYY-MM-DDx.zip`) so existing pre-semver upgrade flows can adopt it directly. After that upgrade lands, the new `check_version.py` with `_to_version()` is in place and future semver-named packs work normally.
- **v1.0.0 (first clean semver release):** No migration step required. Projects on v0.9.0 upgrade automatically via `_to_version()` comparison.

**Version format in `VERSION` and `framework_revision`:** `MAJOR.MINOR.PATCH+<build>`, where `<build>` is the rightmost 4 characters of the lifecycle prefix generated by `lifecycle_id.py --prefix-only` (for example, lifecycle prefix `12tm5` becomes build suffix `2tm5`). Example: `1.0.0+2tm5`.

**Zip artifact filename:** Uses a `.` separator instead of `+` for filesystem safety on all platforms: `wavefoundry-MAJOR.MINOR.PATCH.<build>.zip`. Example: `wavefoundry-1.0.0.2tm5.zip`. The one-time `0.9.0` bridge release is the only exception and keeps the old date-style artifact name.

**Version comparison:** All version ordering uses `packaging.version.Version`. Build metadata is stripped before precedence comparison (per semver spec, build metadata does not affect ordering). Date-shaped `framework_revision` strings (matching `YYYY-MM-DDx`) are mapped to `Version("0.0.0")` so any pre-semver install is always treated as older than any semver release.

**Upgrade hook rewrite:** Hooks that gate on raw string comparison (`ctx.from_version < "2026-05-19a"`) are rewritten to use `packaging.version.Version` with the date-string fallback. This ensures hooks fire correctly for both pre-semver and semver installs during the transition.

**Multiple builds of the same MAJOR.MINOR.PATCH:** When more than one zip exists for the same version number, the upgrader selects by the 4-character build suffix lexicographically descending.

## Consequences

**Positive:**
- Release identifiers communicate compatibility intent via MAJOR.MINOR.PATCH semantics.
- Correct ordering for multi-digit minor/patch versions (e.g., `1.10.0 > 1.9.0`).
- Build traceability preserved via a shortened suffix derived from the lifecycle prefix.
- Date-based installs can adopt the `0.9.0` bridge release through the existing legacy zip-adoption flow, after which semver-aware upgrades take over.
- Single `.` separator in zip filenames is universally safe across macOS, Linux, and Windows.

**Negative / tradeoffs:**
- The `packaging` library must be a declared runtime dependency (`pyproject.toml`) before `check_version.py` can use `Version`. Coordinated with `12tm5-enh python-tool-venv-bootstrap`.
- Existing repos carry date-based `framework_revision`; migration path maps these to `0.0.0` rather than preserving the original date ordering during transition.
- Build metadata (`+2tm5`) is ignored for version precedence per semver spec — two builds of `1.0.0` are semver-equal and must be disambiguated by build suffix sort.

**Constraints imposed:**
- `VERSION` and `framework_revision` must follow `MAJOR.MINOR.PATCH+<build>` format after the first semver packaging run. Bare `MAJOR.MINOR.PATCH` without build metadata is also valid.
- Zip artifact filenames must follow `wavefoundry-MAJOR.MINOR.PATCH.<build>.zip` after the bridge release. The `+` character must not appear in zip filenames. The one-time `0.9.0` bridge release keeps the old date-style artifact name.
- All version comparison must go through `_to_version()` in `check_version.py`, never raw string comparison.
- `packaging` must remain a declared dependency in `pyproject.toml`.
- Any new upgrade hooks must use `_to_version()` for version-gated migration steps.

## Version Bump Policy

Wavefoundry's "public API" is the operator contract: the upgrade flow, config schema, install steps, and the surfaces operators build automation against. Breaking that contract is a MAJOR bump; adding to it compatibly is a MINOR bump; fixing it without visible change is a PATCH bump.

**PATCH** — changes that are invisible to operators in normal operation:
- Bug fixes with no behavior change at the operator surface
- Performance improvements
- Documentation corrections
- Internal refactors with no config, tool, or output changes

**MINOR** — new capabilities that do not require operator action on upgrade:
- New MCP tools, new seeds, new config options (all backward-compatible)
- New framework features that work without any change to existing operator configs or workflows
- Deprecation notices (the deprecated surface still works; removal is a future MAJOR)

**MAJOR** — changes that require operator action to complete an upgrade, or that remove previously-supported behavior:
- Config field removals or renames that break existing `docs/workflow-config.json` files
- Changes to `~/.wavefoundry/` directory structure that require manual migration
- Changes to the zip format or upgrade hook contract that break existing automation
- Removal of an MCP tool or seed that operators depend on
- Python minimum version bump that may require operators to update their interpreter

When in doubt: if an operator who skips reading the release notes would be broken by the change, it is a MAJOR bump.

## Alternatives Considered

| Alternative | Reason rejected |
|-------------|----------------|
| Continue `YYYY-MM-DDx` date strings | Not a public release contract; does not communicate compatibility intent; broken for broader operator adoption |
| Calendar versioning (`YYYY.MM.DD`) | Same lexicographic ordering problem for multi-digit fields; no compatibility signal |
| Start directly at `1.0.0` | Old upgrade scripts use lexicographic comparison; `"1.0.0+<build>"` < `"2026-05-20i"` lexicographically — existing projects cannot upgrade without a bridge |
| Start at `0.1.0` | Would signal a pre-release API; operator confirmed `1.0.0` as the stable public release target after the bridge |
| Retain date string as a second version field | Two parallel version identifiers in the same field add complexity with no ordering benefit |
| Use `+` in zip filenames | Problematic in Windows filesystems and some CLI tools; `.` is universally safe |
| Custom version parser | `packaging.version.Version` is stdlib-adjacent, widely used in the Python ecosystem, and handles edge cases correctly |
| `importlib.metadata` for version comparison | Reads installed package metadata, not file-based version strings; wrong abstraction for Wavefoundry's zip-distribution model |
