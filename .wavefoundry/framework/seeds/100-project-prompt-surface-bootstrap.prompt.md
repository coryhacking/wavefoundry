13. **Module docstring convention** — When generating or updating `docs/prompts/` coding convention docs (or a `docs/contributing/coding-conventions.md` when present), include the following guidance so `kind="code-summary"` chunks in the semantic index are meaningfully populated:

 > **Module-level docstrings:** Every source file should begin with a one-sentence module docstring (or leading comment block for languages without docstring syntax) describing the file's purpose. Format: `"""<One sentence: what this module does.>"""` (Python), `// <One sentence>` (Go/Rust/JS/TS leading comment).  >

 When no coding-conventions doc exists in the target repository, add this rule as a standalone section in the relevant prompt doc that implementation agents read first (e.g. `implement-wave` agent body or `docs/contributing/agent-team-workflow.md`).

Cross-surface rules:

- **`.prompt.md` extension is mandatory for all runnable prompt files** — every file under `docs/prompts/` (including `docs/prompts/agents/`) that is invoked as a command must use the `.prompt.md` extension, not plain `.md`. The only exceptions are `index.md` (the catalog) and `agents/README.md` (the sub-surface index). The indexer assigns `kind="prompt"` via two detection mechanisms: (1) **path-based** — any file under `docs/prompts/` is detected regardless of extension; (2) **extension-based** — any file with the `.prompt.md` suffix is detected regardless of location. The extension makes the convention self-documenting and extends discoverability to runnable prompt files that live outside the `docs/prompts/` tree. Seeds always take `kind="seed"` regardless of suffix. See `008-framework-map.md` **Prompt file naming convention** for the full rule.

**Voice and tone extraction.** When the Split B `content/` subtree is seeded (`docs/design-system/content/voice.md`, `docs/design-system/content/microcopy.json`), populate these files from repository evidence — not from generic best-practice defaults. Extraction sources to check in order of confidence: (1) checked-in content style guides or brand voice docs anywhere in the repository; (2) localization message catalogs (`*.xliff`, `*.strings`, `*.po`, i18n JSON files) for high-frequency UI copy patterns; (3) direct inspection of UI copy in source files for recurring tone signals (formal vs casual, active vs passive, error message framing, empty-state phrasing). When a source file is found, record `sourceType: code`, `confidence: medium` in `source-map.json`. When only scattered copy inspection is available, record `sourceType: visual`, `confidence: low`. When no coherent source exists, emit `null` in `voice.md` sections and record a `content`-category gap at `important` or `critical` severity depending on whether the product has a substantial copy surface. Never generate brand-voice personality or tone adjectives from generic writing advice — that is a `proposed-from-best-practices` proposal and must be labeled as such.
