# 100 - Project Prompt Surface Bootstrap

Intent:

- Generate the repo-local prompt surface that other agents and users interact with after the seed pack is planted.

Canonical public prompt docs to generate in the target repository:

- `docs/prompts/index.md`
- `docs/prompts/install-wavefoundry.md`
- `docs/prompts/upgrade-wavefoundry.md`
- `docs/prompts/plan-feature.md`
- `docs/prompts/create-wave.md`

Tasks:

1. Generate prompt docs that advertise the wave-context phrasing as canonical.
2. Allow legacy alias phrases only as migration aids.
3. Generate or refresh supporting agent-oriented prompt bodies under `docs/prompts/agents/` when the repository keeps checked-in planning/context prompt bodies.
4. Describe the prompt surface as:
   - seed-derived
   - repo-adaptive
   - wave-aware
   - memory-aware
   - persona-aware
5. Explain that artifacts use output contracts rather than rigid templates.
6. Keep the split explicit: `docs/prompts/` is the public shortcut surface, while `docs/prompts/agents/` holds supporting agent-oriented prompt bodies that should not be listed as public commands.
7. Register the prompt surface in `docs/prompts/prompt-surface-manifest.json`, including **`framework_revision`** — a string equal to `.wavefoundry/framework/VERSION` after init or upgrade reconciles the pack (used by `seed-160` **version guard** alongside the same `VERSION` file).
8. Ensure prompt docs reflect the wave/change/item model, wave orchestration contract (wave coordinator as primary organizer per `docs/agents/wave-coordinator.md` when present), add-time relocation of admitted change docs with prepare-time validation/repair, journals, personas, factor-review behavior, and specialist routing when relevant.
9. Describe **`Init wave framework`** / **`Upgrade wave framework`** (legacy: **`Init wave context`** / **`Upgrade wave context`**) as project-understanding refresh paths that reconcile the public prompt surface and `AGENTS.md` with current evidence from the repository — not wrapper-only mechanical updates.
10. Retire or rewrite stale local prompt docs that still advertise only legacy project-context phrasing when the repo has moved to wave-context terminology.
11. Ensure prompt docs route users to the project's actual seeded policies and procedures, including legacy-baseline-derived workflow updates, instead of only pointing back to the shared framework pack.
12. Ensure prompt docs describe the file-level review pause and protected-surface ownership rules for broad framework-maintenance work, routing detailed concurrency expectations through `docs/prompts/agent-routing-concurrency.md`.

Required prompt-surface behaviors:

Each per-prompt rule below names a seeded prompt doc by its short name (e.g. `implement-wave`). The rule applies to the public surface `docs/prompts/<name>.md` and — when it exists — to the agent-oriented body at `docs/prompts/agents/<name>.md` jointly; where a rule is public-only or agent-body-only, that scope is called out explicitly.

Per-prompt rules:

- **implement-feature + implement-wave**: enforce the repository-code stage gate as the first guardrail — do not edit repository code until the change is documented, admitted into the active wave, and that wave has a clean `Prepare wave` / `Ready wave` pass as the immediately preceding lifecycle step; if any staging step is missing, stop and route back to `Plan feature`, `Create wave`, `Add change to wave`, or `Prepare wave`. Additionally require agents to confirm `AGENTS.md` **Implementation guard (product code)** before editing product implementation paths when that section exists (and to run or reconcile `Prepare wave` / `Ready wave` for wave execution per `seed-180`). After substantive `docs/` edits, instruct agents to verify the docs gate with MCP **`wave_validate`** (and **`wave_garden`** when metadata timestamps need refresh) when the Wavefoundry MCP server is available; use **`.wavefoundry/bin/docs-lint`** / **`.wavefoundry/bin/docs-gardener`** only as **CLI fallback** (no MCP) or to interpret hook failures.
- **prepare-wave**: treat the readiness evaluation as part of the repository-code stage gate — validate that admitted change docs are already in `docs/waves/<wave-id>/`, repair staged-only placement drift before recording a ready verdict, and fail loudly on duplicate staged + wave copies; repository-code implementation may not begin until the target change is documented, admitted into the wave, this readiness pass is clean, and the required readiness-time review lanes are selected, run, or explicitly recorded. Require **`product-owner`** for product-impacting waves as **early gap detection** before implementation and **`product-owner` again** when the admitted change set changed since the last clean **`Prepare wave`** (unless **product-owner: N/A** with rationale covers the delta). Require **`qa-reviewer` at minimum** whenever the admitted change set includes a **bug** (`change-id` kind `bug`) or other **product defect fix** — per `docs/contributing/agent-team-workflow.md`, `docs/contributing/review-and-evals.md`, and `docs/workflow-config.json` `review_policies.require_qa_reviewer_for_bug_fixes`; record the lane in **Review checkpoints** before closure, and do not treat "small patch" or single-implementer scope as exempt unless the operator records an explicit scoped waiver in the wave or change doc. Include an **AC priority check** step: after confirming readiness, the wave coordinator categorizes each admitted change's acceptance criteria as required / important / nice-to-have / not this scope and records the prioritization in the wave record or admitted change doc before implementation begins — one pass, bounded to admitted ACs; use a stable heading (recommended: **`## AC priority`** on the change doc) so **`Review wave`** can find it. Then interrogate **required** and **important** ACs until each classification is explicitly justified: ask follow-up questions as needed (multi-turn if necessary) until the reasoning is clear; **nice-to-have** items are accepted without interrogation; **not-this-scope** items need only a one-line disposition (deferred to a follow-on or permanently out of scope). The interrogation step is bounded to admitted ACs and must not extend **`Prepare wave`** into a full re-planning exercise. **AC priority — product-owner acknowledgment (cross-project):** on **product-impacting** waves (any **product-owner** not **N/A**), before **merging** implementation to the project's main integration line (default branch, release branch, or operator-named integration target — adapt to the repository's workflow), the **`product-owner`** must **explicitly acknowledge** the recorded AC priority table: confirm **required** items are blocking for closure, **not-this-scope** deferrals are acceptable, and **important/nice-to-have** are not mistaken for **required** — record **Verdict**, **date**, and a **one-line summary** in the wave record **`## Review checkpoints`** (stable bullet title, e.g. **`product-owner` — delivery scope sign-off**). If the AC priority table **changes** during implementation, re-run acknowledgment **before merge**. Single-operator projects may use the same record with operator hat **product-owner**. **AC priority — review-wave reconciliation (cross-project):** at **`Review wave`** (including closure-time review), reconcile the AC priority table against **delivered** behavior and docs; update the table if reality diverged. The **`qa-reviewer`** must confirm every **required** row has verification evidence (automated test, manual matrix, or documented exception) or record **deferred** with rationale — record under **`## Review checkpoints`** (e.g. **`Review wave` — AC priority reconciliation**) before **`Close wave`**. **`seed-180`** implement loop: do not treat the wave as **merge-ready** until any required **product-owner** delivery scope sign-off is recorded when this rule applies.
- **add-change-to-wave**: note that admission is a hard stage gate for later code work — if the change is not admitted, implementation prompts must stop instead of editing code. In add/apply mode, move the active change doc into `docs/waves/<wave-id>/` immediately; repeated calls should be safe when the doc is already in the target wave, and duplicate staged + wave copies should be treated as an explicit conflict. Require **`product-owner`** review of the **admission delta** when the admitted work is a **feature** or otherwise shifts product behavior, UX, or acceptance expectations (or record **product-owner: N/A** with rationale for non-product admissions); the next **`Prepare wave`** must validate or repair placement and reconcile the expanded admit set per `docs/contributing/agent-team-workflow.md`.
- **plan-feature**: require that repository-code work be explicitly staged for later delivery (wave admission required before implementation begins) and define the readiness evidence that `Prepare wave` must validate before implementation starts. Require a **`## Affected architecture docs`** section on consolidated change documents when the work crosses **modules**, **integration or trust boundaries**, **primary data/control paths**, or **test/release seams** — list which of `docs/ARCHITECTURE.md`, `docs/architecture/{current-state,domain-map,layering-rules,cross-cutting-concerns,data-and-control-flow,testing-architecture}.md`, or `docs/architecture/decisions/*` need updates during or after delivery (or **`N/A`** with one-line rationale when the change is confined to a single module with no boundary, flow, or verification-topology impact). Require framework/prompt-surface maintenance plans to name intended file edits, protected surfaces, and read-only vs write-owning lanes before execution begins. After creating a plan, note that **`Interrogate this plan`** / **`Stress-test this plan`** (see `docs/prompts/interrogate-plan.md` and `.wavefoundry/framework/seeds/175-interrogate-plan.prompt.md`) is available as an optional stress-testing shortcut to walk every unresolved decision branch and confirm planning depth before admission or implementation begins.
- **interrogate-plan**: implement the interrogation contract from `seed-175` (`175-interrogate-plan.prompt.md`): given a change doc or wave record as context, walk every unresolved decision branch one question at a time, provide a recommended answer and cite the supporting project resource, self-answer without operator input when derivable from source code, specs, architecture docs, references, or agent docs, and stop when all branches in Requirements, Acceptance Criteria, and Scope are resolved; support `--batch` mode for a full question list rather than interactive back-and-forth; this is an optional stress-testing tool, not a required lifecycle gate.
- **index.md** (public-only): include an **Operating Rules** section (or equivalent) that references `.wavefoundry/framework/seeds/020-run-contract.prompt.md` as the authoritative run contract and carries the key behavioral rules from that contract inline — at minimum: scoped-work triage (classify by risk, blast radius, and lifecycle impact; **lightweight** means compact output **within** the lifecycle checkpoints that apply per `AGENTS.md` — not a skipped document/admit/`Prepare wave` path when those gates apply; complex/high-stakes and always-gated work requires full reasoning depth and lifecycle gates), brownfield pattern detection and compliance (detect dominant patterns before implementing, follow them, surface significant problems for operator approval, no silent divergence or silent improvement), surface assumptions explicitly, prefer the smallest correct change (after pattern obligations), diagnose before retrying, prefer one clarifying question over a wrong assumption, and verify changes actually solved the problem after the fact; backfill when missing on upgrade. Usage notes must describe the full lifecycle sequence required before code edits (document → admit → `Prepare wave`) and mention the implementation guard when `AGENTS.md` includes it.
- **init-wave-context + upgrade-wave-context**: tell agents that **`AGENTS.md` → Start Here** includes **Git commits (operator-owned)** per `050` when that section exists — agents must not run `git commit` unless the operator explicitly requests it in the **current** request; default handoff is diff plus a suggested commit message for the operator — consistent with `docs/contributing/build-and-verification.md` **Git commits** when that subsection is seeded.
- **upgrade-wavefoundry**: keep inventory/drift-detection subagents read-only, require a concise file-level upgrade plan before broad framework/prompt-surface edits, and tell agents to update an existing canonical doc in place rather than creating a parallel file when a topical home already exists. Describe **step 0** distribution-zip adoption at the repository root when `seed-160` does — lexicographic zip selection, `unzip -o`, immediate `render_platform_surfaces.py`, then full reconciliation — and include a **Verification checklist** (or equivalent) subsection pointing to **`docs/contributing/build-and-verification.md`** (**Wave framework pack upgrade verification**) for the ordered operator commands; tell **agents** to prefer MCP **`wave_garden`** then **`wave_validate`** for the docs gate when MCP is attached, with **`.wavefoundry/bin/docs-gardener && .wavefoundry/bin/docs-lint`** as the **operator/CI** path when MCP is not attached; when `docs/prompts/package-wavefoundry.md` is seeded, it must stay consistent with that zip-first upgrade path.
- **package-wavefoundry** (public-only, when present): describe `.wavefoundry/framework/scripts/build_pack.py` accurately — default zip date is **today** (local ISO); letter suffix is the **next letter after the maximum suffix already present** for that date in the output directory (not the first missing `a`–`z` gap when a higher letter zip exists); before the final zip build, `docs/prompts/prompt-surface-manifest.json` **`framework_revision`** must be updated to the revision being packaged; the script then stamps `.wavefoundry/framework/VERSION` to `<date><letter>` before building the zip; `--date` is for tests or exceptional re-builds only.
- **implement-wave**: define implementation as a coordinator-owned implementation-review loop, not a pure coding phase — required reviewer lanes chosen by readiness must participate during execution, blocking findings return the wave to implementation, and the wave is not execution-clean until required review lanes are clean or explicitly deferred with recorded rationale. Include the MCP-first docs gate guidance from the **implement-feature + implement-wave** rule above ( **`wave_validate` / `wave_garden`** before bin CLI).
- **agent-routing-concurrency** (public-only): describe read-only vs write-owning lanes, protected-surface serialization, and coordinator reread verification after delegated writes.
- **index.md** and agent-routing docs: when the repository enables specialist roles, explain the distinction between universal specialists, archetype specialists, and repo-local specialists, and route readers to the canonical specialist catalog rather than dumping every optional role into the public command table.
- **close-wave**: require **docs-contract / spec review** at wave closure when `docs/specs/*.md` or other canonical runtime-behavior docs changed during the wave, with a **Review checkpoints** entry in `docs/waves/<wave-id>/wave.md` (or **Docs-contract review: not applicable** with rationale); reference `docs/contributing/review-and-evals.md` (**Wave closure**) and stay aligned with `seed-190`. Closure review is mandatory even when no docs-contract lane is triggered — required review lanes from readiness and execution must be reconciled and recorded in **Review checkpoints** before the wave can close. Explicitly block closure until chronology and closure-artifact reconciliation are complete — update `Completed at` plus wave/change status fields, reconcile journal updates/distillation, promote durable memory updates to `docs/references/project-context-memory.md` (and other canonical docs when applicable), and clear or refresh `docs/agents/session-handoff.md`. Include the terminal **docs gate** step: **agents** prefer MCP **`wave_garden`** then **`wave_validate`** ( **`wave_audit`** optional); **operators / CI / no MCP** use **`.wavefoundry/bin/docs-gardener`** then **`.wavefoundry/bin/docs-lint`** — same split as `seed-190` / **`finalize-feature`**.
- **review-wave**: tell the coordinator that **`code-reviewer` is not optional** when `docs/contributing/agent-team-workflow.md` / readiness evaluation requires it or when the admitted change set touches non-trivial product logic; the review pass must include explicit **branch-complete** and **re-entrant** checks for any per-key mutable state the change touches (see `seed-190` task 7 **code review** bullet). Seeded `qa-reviewer` expectations must include **multi-step** verification (state across repeated calls or routine steps) when such state exists, or record deferral and residual risk. Include an **AC scope gap check** step: after confirming required ACs are met, surface important/nice-to-have items not in admitted scope that would add value and confirm not-this-scope deferrals — output a short prioritized list so the operator can decide on follow-on scope before closure. One lightweight pass, bounded to admitted ACs. When admitted change docs include **`## AC priority`** (or equivalent from **`Prepare wave`**), the review pass must **reconcile** that table to **shipped** behavior (update the table if scope shifted) and must include **`qa-reviewer`** attestation that each **required** row has verification or a recorded deferral — see **`prepare-wave`** **AC priority — review-wave reconciliation** in this seed.

Cross-surface rules:

- keep the public surface limited to the intended user-facing prompts
- make wave-operator actions first-class public prompts instead of burying them only in umbrella feature prompts
- ensure there is a first-class readiness gate prompt that can run explicitly and can also be auto-invoked before implementation
- keep checked-in agent-oriented prompt bodies under `docs/prompts/agents/` when they are part of the repo-local seeded surface
- keep internal helpers out of the normal prompt index
- preserve legacy aliases only as migration aids
- ensure prompt docs reference canonical repo-local docs rather than only the shared pack
- ensure prompt docs explain init/upgrade planting behavior clearly enough for seeded repositories to use them consistently
- ensure prompt docs describe enough project-specific procedure context that agents can find the right canonical operating docs without guessing

## Design-System Extraction — Prompt Surface Note

Design-system extraction (seeded under `docs/design-system/` by `seed-040` task 14 and `seed-010` step 8) does **not** introduce new public shortcut phrases. The extraction contract is activated through the existing `Init wave framework` and `Upgrade wave framework` lifecycle paths. No additional operator commands are needed.

When generating or refreshing `docs/prompts/index.md`, do **not** add a standalone `Extract design system` or `Design extraction` public shortcut. Instead, ensure the index notes that design extraction runs as part of install/upgrade backfill when `docs/design-system/` is present or seeded. Operators who want to trigger a clean re-extraction manually should follow the rollback path documented in `docs/architecture/design-system.md` (when present) and the `meta`-category `gaps.md` entry protocol defined in `seed-160`.

**Voice and tone extraction.** When the Split B `content/` subtree is seeded (`docs/design-system/content/voice.md`, `docs/design-system/content/microcopy.json`), populate these files from repository evidence — not from generic best-practice defaults. Extraction sources to check in order of confidence: (1) checked-in content style guides or brand voice docs anywhere in the repository; (2) localization message catalogs (`*.xliff`, `*.strings`, `*.po`, i18n JSON files) for high-frequency UI copy patterns; (3) direct inspection of UI copy in source files for recurring tone signals (formal vs casual, active vs passive, error message framing, empty-state phrasing). When a source file is found, record `sourceType: code`, `confidence: medium` in `source-map.json`. When only scattered copy inspection is available, record `sourceType: visual`, `confidence: low`. When no coherent source exists, emit `null` in `voice.md` sections and record a `content`-category gap at `important` or `critical` severity depending on whether the product has a substantial copy surface. Never generate brand-voice personality or tone adjectives from generic writing advice — that is a `proposed-from-best-practices` proposal and must be labeled as such.

Guardrails:

- Keep the public command surface small and memorable.
- Do not expose internal helper prompts in normal repo-local prompt indexes.
- Do not collapse agent-oriented prompt bodies back into a root-level `agent-prompts/` folder or the public prompt index.
