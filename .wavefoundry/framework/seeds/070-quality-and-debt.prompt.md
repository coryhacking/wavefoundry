# 070 - Quality And Debt

Intent:

- Establish quality posture and supporting docs that help wave planning, review, and upgrade decisions stay grounded.

Tasks:

1. Create or update quality baseline docs.
2. Create or update reliability, security, and performance posture docs. **Name performance and reliability hotspots using the same module/process boundaries** established in `seed-060` (`docs/architecture/current-state.md`, `domain-map.md`) so reviewers can cross-reference without conflicting terminology. When `seed-030` / `seed-031` recorded unpinned container image tags (for example `latest`) in CI configs, Helm values, or Kubernetes manifests, classify them as a reliability / reproducibility risk in the posture docs and link to the specific references so the fix surface is obvious.
3. Create or update debt tracking or execution-plan support docs when any of the following are true: unresolved debt items were discovered during the inventory or domain-boundary phases; `docs/references/tech-debt-tracker.md` does not yet exist; or the wave plan identified items explicitly deferred for later resolution. When none of these conditions hold, skip this task rather than creating an empty file.
4. Record review triggers and sensitivity signals that later prompts can reuse (security, performance, reliability, and factor-oriented discovery should stay aligned with `docs/repo-profile.json` factor signals where applicable).
5. Record whether factor-oriented review is relevant when `docs/repo-profile.json` includes a `factor_review` block: for each factor marked `applicable`, confirm that the matching agent file will be generated by `seed-050` and that its review questions are grounded in project-specific evidence from this inventory. For factors marked `partial`, record what would make them fully applicable. For factors marked `not-applicable`, record the rationale so future evaluations can reconsider if the project shape changes. When `factor_review` is absent from `docs/repo-profile.json`, skip this task and flag as a gap in `docs/missing-docs.md`.

Required semantics:

- main quality risks
- review-sensitive surfaces
- known debt that affects execution or upgrade risk
- performance and reliability hotspots
- factor-review applicability and relevant factors when factor review is appropriate

Guardrails:

- Keep these docs evidence-based.
- Promote durable quality/process expectations into canonical docs, not journals.
